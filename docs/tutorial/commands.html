Session Lifecycle

Session Lifecycle

These commands form the daily rhythm. Every session starts with /stand-up and ends with /wrap. Everything in between is yours to direct.

/stand-up DOE Starts your session and shows you where things stand.

When to use it: At the beginning of every session. It's always your first command.

/stand-up has two modes and picks the right one automatically:

  • Kick-off mode (no session running): Starts the session clock, reads STATE.md and todo.md, and shows a status card with current progress, blockers, a plan, and coaching tips.
  • Status mode (session already running): Shows the same status card but doesn't reset anything. Read-only and safe to run any time.

You don't need to remember what you were doing last time — the stand-up reads STATE.md and reconstructs your context automatically.

What each section means

  • FEATURE — The active feature from todo.md, with its type tag ([APP] or [INFRA]) and version range.
  • PROGRESS — Visual progress bar. Filled blocks are completed steps, empty blocks are remaining.
  • CONTRACT — Whether the next step's contract is valid and how many auto vs manual checks it has.
  • BLOCKERS — Active issues from STATE.md that might affect work. Only appears if there are blockers.
  • DOE KIT — Framework version. "synced" means up to date. "* push" or "* pull" means updates are pending.
  • SIGN-OFF — How many features are in Awaiting Sign-off and how many manual items still need your testing. Only appears when items are pending.
  • PIPELINE — How many features are queued in ROADMAP.md vs todo.md. Nudges you to promote or scope new work.
  • Last session summary — What happened last time, pulled from STATE.md so you don't have to remember.
  • PLAN — What Claude recommends doing this session. You approve, adjust, or redirect before work starts.
  • FOCUS — Coaching insights from your session stats. Patterns, trends, and suggestions to keep you on track.
  • Model row — Which AI model and thinking level you're using. Helps you decide if you need to switch for the task at hand.
/crack-on DOE Resume after a break — refamiliarises and starts building immediately.

When to use it: When context was lost (you had to /clear, came back after a break, started a new conversation) and you want Claude to resume building without a full briefing. Not for fresh sessions — use /stand-up for those.

How it works:

  1. Starts the session clock, reads CLAUDE.md, STATE.md, todo.md, and learnings.md.
  2. Finds the next incomplete step and validates its contract.
  3. Shows a compact bordered card summarising the current state.
  4. Immediately starts building — no sign-off needed.
  5. Builds one step, runs [auto] verification, commits, then stops and shows what was done.

If an auto-check fails, Claude attempts to fix the issue (up to 3 tries) before asking for help. For tasks with [manual] criteria, Claude notes what you need to check but doesn't stop and wait.

ⓘ
/stand-up vs /crack-on: /stand-up shows a full briefing and waits for your approval before building. /crack-on shows a compact card and starts building immediately. Use /stand-up for fresh sessions; /crack-on to resume after interruptions.
/sitrep DOE Quick mid-session progress snapshot. Read-only, completely safe.

When to use it: Whenever you want to see where things stand without changing anything. Run it as often as you like — no side effects.

How it works: Reads the current feature's todo.md steps and shows which are done, which is current, and what's coming next. Think of it as a progress bar for your feature.

/wrap DOE Closes your session. Records what happened so the project remembers it next time.

When to use it: Before ending any session. Always. This is how continuity works — if you skip the wrap, the next session's /stand-up won't know what you did.

How it works:

  1. Updates STATE.md with the current position (what was done, what's next, any blockers).
  2. Records any new learnings to learnings.md.
  3. Calculates session stats: commits, lines changed, duration, and your streak.
  4. Generates a formatted summary of the session.

Think of /wrap as saving your game. The state is written to files that persist between sessions.

Quality & Review

Your testing and inspection toolkit. These commands help you verify that things are correct — from individual task contracts to project-wide health.

/audit DOE Runs automated checks across the entire project and reports what needs attention.

When to use it: Before commits, before releases, or as a periodic confidence check.

What it checks:

  • Governed doc freshness — Are documents like data-governance.md up to date?
  • Task format — Do all tasks in todo.md have proper contracts?
  • Roadmap consistency — Do status tags match reality?
  • Git status — Any uncommitted changes or loose ends?
  • Framework integrity — Are all DOE components present and correctly configured?

What to expect: A checklist with PASS, WARN, or FAIL for each category. Warnings are things to look at when convenient. Failures should be fixed before proceeding.

/agent-verify DOE Runs all Verify: patterns from the current task's contract.

When to use it: During and after building a task, to confirm work meets its contract.

How it works: Every task in todo.md has criteria marked [auto] (machine-checkable) and [manual] (needs human eyes). Each [auto] criterion has a Verify: pattern. This command runs all of them.

Verify pattern types:

PatternWhat it does
run:Executes a command, checks exit code
file: ... existsChecks a file exists
file: ... containsChecks a file contains specific text
html: ... hasChecks an HTML file for an element
/fact-check DOE Reads a document and checks whether its claims match the actual codebase.

When to use it: After major refactors that might have made documentation stale, or before sharing docs externally.

How it works: Claude reads the target document, identifies factual claims (function names, file paths, behaviour descriptions, data formats), then checks each one against the codebase. Claims that don't match reality get flagged and corrected directly in the file. Review the changes before committing if you want to verify them.

/review DOE Adversarial code review — looking for problems, not compliments.

When to use it: Before shipping a feature, before merging a branch, or when you want a critical second opinion. Especially valuable for code written quickly.

What it checks:

  • Bugs — Logic errors, off-by-one mistakes, unhandled edge cases
  • Security issues — Exposed credentials, injection vulnerabilities
  • Performance problems — Unnecessary loops, redundant API calls
  • Style issues — Inconsistent naming, unclear variables

Think of it as a code review from a colleague who cares about quality but doesn't care about your feelings.

/snagging DOE Runs automated tests, then generates an interactive HTML checklist with results card and manual check sections.

When to use it: After a feature is code-complete and all [auto] criteria pass. Runs the automated test suite first, then presents manual items for human testing.

What it generates: A standalone HTML page at docs/[feature-name]-manual-tests.html with an automated results card (status badge, metric tiles, expandable details for Playwright/Lighthouse/a11y/health check results) and manual check sections with three-state checkboxes (untested/pass/fail), copyable console commands, setup prerequisites, a test timer, localStorage persistence, and export buttons for pasting results back to Claude. Supports dark mode toggle and baseline updates.

How it works: First runs the automated test suite if an orchestrator exists — Playwright browser tests, Lighthouse performance/SEO audits, accessibility checks, and health check scripts. Then reads todo.md, extracts [manual] items for the current feature, splits multi-sentence items into individual checks, detects console commands, and traces code for known edge cases. Results from both automated and manual phases are combined into a single HTML page. The generated checklist is your sign-off workflow — review automated results, test manual items, export failures, paste back, fix, regenerate, repeat until all pass.

/report-doe-bug DOE Triage-first bug reporter for the DOE framework itself. Gathers context, checks fixes, detects user error, and files a GitHub Issue only if needed.

When to use it: When you hit a bug in the DOE framework — a command crashes, a hook blocks valid actions, an execution script fails on valid input, or documentation is wrong.

How it works: Runs a 5-phase triage pipeline. First gathers your description and Claude’s reconstruction of what happened, plus environment data (DOE version, OS, Node, Python). Then checks if the bug is already fixed in a newer version (routes you to /pull-doe). Then assesses whether it’s a framework bug or a usage mistake (routes you to the relevant tutorial page). Then searches existing GitHub issues for duplicates. Only if it passes all three gates does it sanitise the report (stripping API keys, secrets, and project-specific paths) and file a structured GitHub Issue with labels. Falls back to a local markdown file if gh CLI is not available.

/test-suite DOE Runs the project's accumulated test suite and reports results.

When to use it: Before committing, after refactoring, or whenever you want to check that existing functionality still works.

How it works: Executes all test scripts accumulated over time. Each passing task's Verify: patterns often get promoted into permanent tests, so the suite grows as the project grows. A failing test tells you exactly what broke, so you can fix it before the problem reaches users.

Planning & Ideas

These commands help you think about what to build, not just how to build it.

/scope DOE Takes a fuzzy idea and turns it into a clear, buildable brief through guided conversation.

When to use it: When you have an idea but haven't figured out the details.

How it works: The scoping session has three phases:

  1. Explore — What problem are we solving? Who has this problem?
  2. Define — What does the solution look like? What data does it need?
  3. Bound — What's in scope and what's not? What's the minimum viable version?

The output is a scoping document that can become a ROADMAP.md entry or a set of todo.md tasks. The session doesn't build anything — it clarifies what to build.

/pitch DOE Reads the project state and generates strategic suggestions for what to build next.

When to use it: When you've finished a feature and are wondering what's next, or when you want Claude to spot opportunities you might have missed.

How it works: Claude reads across the project — ROADMAP.md, STATE.md, learnings.md, the codebase — and identifies gaps, natural extensions, or improvements that would add genuine value.

Pitches are suggestions, not commitments. Say "add it" to put it on the roadmap, "this is important" to flag it, or just ignore it.

/roast DOE Honest, humorous critique of your codebase — looking for the embarrassing bits.

When to use it: When you want brutal honesty about code quality, as a sanity check before a demo, or just for entertainment.

What it looks for: messy code that works but shouldn't, forgotten TODOs and FIXMEs, architectural smells, inconsistencies (doing the same thing three different ways), dead code, and overly clever solutions.

Constructive underneath the humour — each observation points to something you could actually improve. Nothing mean-spirited, but nothing sugarcoated either.

Advanced: Parallel Work

Run multiple Claude Code sessions in parallel, each working on a different task. Most users won't need this initially, but it shines when you have a batch of independent work.

/agent-launch DOE Sets up a wave — scans tasks, finds parallel candidates, assigns each to a terminal.

When to use it: When you have 2+ independent tasks ready to build. Three 20-minute tasks take 20 minutes in a wave instead of 60 sequentially.

How it works:

  1. Reads todo.md and identifies incomplete tasks.
  2. Checks which tasks are independent (no shared files, no output dependencies).
  3. Validates contracts for each task.
  4. Creates a git worktree for each task (an isolated code copy).
  5. Tells you which terminals to open and what to run in each.
ⓘ
Key constraint: Tasks that edit the same file can't run in parallel. Wave agents must not edit shared files (todo.md, CLAUDE.md, learnings.md, STATE.md). Merging happens from the coordinator terminal after all agents finish.
/agent-status DOE Dashboard of the current wave — which terminals are active and how far along.

When to use it: During an active wave, from the coordinator terminal. Check in periodically to see how things are going.

What to expect: A status card showing each terminal's assigned task, current status (building, verifying, complete), last commit, and the merge order. The merge order is determined automatically to minimise conflicts — tasks touching fewer files merge first.

Maintenance

The plumbing — keeping the framework updated, checking what's installed, and managing the connection between your project and the DOE Starter Kit.

/pull-doe DOE Pulls the latest DOE Starter Kit updates into your project.

When to use it: When the starter kit has been updated with new commands, bug fixes, or improved hooks.

How it works: Checks your current version against the latest, shows a changelog of what changed, and applies updates safely without overwriting your project-specific customisations. If there's a conflict, you'll be shown it and asked how to resolve it.

/commands DOE Shows all available slash commands, their status, and version information.

When to use it: To see what commands are available, check if something is installed, or troubleshoot a missing command.

How it works: Scans the .claude/commands/ directory and checks each command file. If a command is missing or broken, the output tells you what's wrong and how to fix it — usually /pull-doe resolves it.

/codemap DOE Generates a structured index of the project — files, functions, and relationships.

When to use it: When joining a new project for the first time, or after significant structural changes. The codemap helps Claude understand the codebase faster in future sessions.

How it works: Scans the project's files, reads their contents, and builds a map of what each file does, key functions and signatures, dependencies between files, and entry points and data flow. Written to .claude/codemap.md.

The file is for Claude's reference, but it's human-readable too — useful for onboarding or understanding unfamiliar parts of the project.

Default Commands

These commands come with Claude Code itself — they’re available in any project, whether or not you use DOE.

/clear Default Clears conversation history and resets context. Use when switching topics or when context gets polluted.
/help Default Shows available commands and usage information.
/model Default Switches the active model (e.g. Opus, Sonnet, Haiku) mid-session.
/memory Default Manages persistent memory — facts Claude remembers across sessions.
/context Default Shows current context window usage and what’s loaded.
/compact Default Compresses conversation history to free up context space without losing key information.
