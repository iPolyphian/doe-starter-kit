Testing & Quality

DOE has a multi-layer quality system that catches bugs automatically so you can focus on building. Most of it runs without you doing anything.

ⓘ
Nothing to memorise. Claude tells you what ran, what passed, and what needs your attention at every step transition. If Claude doesn't ask, everything passed.

The Three Layers

Every DOE project checks code quality in three fundamentally different ways. Together they cover almost every class of bug.

Deterministic

ESLint + type checking. Same result every time. Runs automatically on every commit.

Probabilistic

Playwright, Lighthouse, axe-core run your app and measure what happens. Catches broken pages, perf regressions, a11y violations.

Empirical

Manual snagging + /code-trace. Human judgment catches UX issues, visual bugs, and logic gaps that tools miss.

Why three layers?

Each layer catches a different category of problem that the others miss:

  • Deterministic checks catch typos, undefined variables, and unused code. They're fast, cheap, and mechanical. The same lint rule fires the same way every time.
  • Probabilistic checks catch runtime and integration failures — a page that loads but throws a console error, a performance regression, an accessibility violation introduced by a layout change. These tools run your app and measure what happens.
  • Empirical checks catch UX and logic problems that tools can't see — a flow that feels wrong, a visual glitch on a specific device, a feature that technically works but confuses users. These require human judgment.

The filter funnel

Think of the three layers as a funnel. Bugs enter at the top; each layer catches a different class before it reaches users.

Your Role

The quality system is designed to run itself. Here's what you actually contribute:

  • You define what "working" means
    Contracts in todo.md specify what a passing feature looks like. Without these, Claude can't verify its own work. They take 30 seconds to write and prevent hours of rework.
  • Claude installs and configures tools
    ESLint, Playwright, the regression suite — Claude sets these up when starting a new project. You approve once, and they stay configured. You never touch the config files directly.
  • Tests run automatically — results appear in the snagging checklist
    When a feature completes, the snagging checklist shows automated results as metric tiles you can read at a glance. Green tiles you can ignore. Red failures come with a plain-English explanation and a fix recommendation. Manual checks appear below for things only a human can judge.
  • When something breaks
    Claude explains what failed, why it likely happened, and what to try first. For anything Claude can't diagnose automatically, see the Troubleshooting page — including Maestro issues, CI failures, and tests pass but app broken.

Commands

These quality commands are available in any DOE session. You rarely need to invoke them manually — Claude calls them at the right moments — but you can run any of them yourself at any time.

/code-trace [module]
Trace a module for logic bugs

Claude reads the specified module and traces every code path looking for edge cases, incorrect returns, and logic errors. Replace [module] with a filename or a natural description like "the import script".

/code-trace --integration
Check cross-module data flow

Checks how modules pass data to each other — mismatched field names, type mismatches, values that transform in unexpected ways as they move through the pipeline.

/test-suite
Run the regression suite

Runs all automated tests that have been accumulated for the project. Each test checks a behaviour that was previously verified as correct. A pass means nothing has regressed since the last run.

/snagging
Run test suite + generate snagging checklist

Runs the full automated test suite (Playwright browser tests, Lighthouse performance audit, accessibility scan, health check), then generates an interactive HTML checklist. Automated results appear as metric tiles at the top; manual checks follow below a "YOUR REVIEW" divider. See The Snagging Checklist for details.

/review
Adversarial code review

Claude adopts an adversarial stance and reviews the recent changes looking for bugs, security issues, and design problems. Useful after complex data-layer work or before a significant release.

/audit
Claims truthfulness check

Verifies that all data-backed claims in the project (charts, statistics, summaries) match the underlying source data. Catches stale numbers and copy-paste errors before they reach users.

The Snagging Checklist

When a feature is code-complete, Claude runs /snagging to produce an interactive HTML checklist. This is where all quality results converge into one page you can review.

How it works

  1. Claude runs the automated test suite (Playwright, Lighthouse, accessibility, health check) — takes about 30–60 seconds.
  2. Results are collected into a JSON file and passed to the checklist generator.
  3. The checklist HTML opens in your browser with two distinct sections: automated results at the top, and manual checks below a "YOUR REVIEW" banner divider.
  4. You review the automated tiles at a glance, then work through the manual items. Mark each manual check pass or fail, click "Copy Results", and paste back to Claude.

Automated results tiles

The top of the checklist shows four metric tiles summarising what the machines found. Each tile has its own pass/warn/fail status — if one tool crashes, the others still report independently.

Tile What It Shows Thresholds
Browser Tests Pass/total count (e.g., "32/32") plus the number of routes configured Green = all pass. Red = any failures.
Visual Regression Number of screenshot diffs detected (e.g., "0 diffs") Green = 0 diffs. Red = diffs found (with page names listed).
Accessibility New violation count vs baseline (e.g., "0 new, 3 known") Green = no new violations. Amber = known violations older than 30 days. Red = new violations detected.
Performance Lighthouse score with delta from baseline (e.g., "78 (+0)") Green = within ±5 of baseline (noise-adjusted). Amber = ±6–10. Red = more than ±10.

A Health Check tile appears as a 5th tile when warnings or failures are found (e.g., stubs or TODOs in the code). When clean, health check results are available in a collapsible detail section only.

Above the tiles, a status badge summarises the overall state: ALL PASS (green), WARNINGS (amber), or FAILURES (red).

ⓘ
Dark mode. The snagging checklist includes a dark mode toggle in the top-right corner, just like these docs. Your preference is remembered across sessions.

Baselines: why they exist

Every quality metric needs a reference point. Without one, the system can only tell you "your Lighthouse score is 78" — not whether that's good or bad, better or worse than yesterday. Baselines solve this by recording a "known good" state that you've approved. Every future test run compares against that baseline and reports the delta.

"At baseline" means the current measurement matches what was last approved. A green tile in the snagging checklist means the metric hasn't regressed from its baseline.

Baseline types

  • Visual screenshots — pixel-level reference images for each page. Diffs flag layout shifts and CSS regressions.
  • Lighthouse scores — the approved performance score. Small fluctuations (±5 points) are treated as noise; bigger drops are flagged.
  • Accessibility violations — the count and types of known violations. New violations are flagged immediately; fixing one prompts you to update the baseline.

When to update baselines

Update baselines after intentional changes: a page redesign, a new page added, an accepted performance trade-off, or a fixed accessibility issue. Never update baselines to silence a regression you haven't investigated.

Tell Claude which baselines to update:

  • --update-baselines — update all baselines (visual, Lighthouse, accessibility)
  • --update-visual — accept current screenshots as the new reference
  • --update-lighthouse — accept current Lighthouse score as the new baseline
  • --update-a11y — accept current accessibility violations as the known baseline

After any baseline update, the test suite automatically re-runs and regenerates the checklist so you see clean results immediately — no need to run /snagging twice.

✓
Baseline improvements are detected automatically. If you fix an accessibility violation and the count drops below the baseline, the checklist flags it: "violations reduced from 3 to 2 — run --update-a11y to update baseline." No manual bookkeeping needed.
⚠
Don't update baselines blindly. If a test fails and you don't know why, investigate first. Updating the baseline to make the red go away just hides the problem. See Troubleshooting for common failure scenarios.

Mobile Testing

DOE automatically detects mobile projects (React Native, Expo, Flutter) from your tests/config.json projectType field. When a mobile project is detected, the quality stack adapts:

  • Maestro replaces Playwright for UI testing — native device flows instead of browser automation
  • Bundle size tracking monitors your app binary and alerts on >5% growth between builds
  • Static a11y linting in /code-trace checks for missing accessibilityLabel (React Native), Semantics widgets (Flutter), and undersized touch targets
  • Mobile-specific snagging sections appear in the checklist: Device Conditions, Accessibility Quick Check, and Network Resilience

Getting Started with Maestro

Maestro is a mobile UI testing framework — think of it as Playwright but for native apps. It drives real iOS simulators and Android emulators, tapping buttons, entering text, and asserting that elements appear on screen. DOE uses it whenever your project type is react-native, expo, or flutter (auto-detected from tests/config.json).

Installation

The easiest way to install Maestro is through the /snagging command, which runs --bootstrap automatically on first use. This installs the Maestro CLI and creates a template flow directory. You can also install manually:

curl -Ls "https://get.maestro.mobile.dev" | bash

The --bootstrap flag does two things for mobile projects:

  1. Installs the Maestro CLI (if not already present)
  2. Creates a .maestro/ directory with a template app-launch.yaml flow

How flows work

Maestro flows are YAML files in your project's .maestro/ directory. Each file describes a sequence of UI interactions. Here's the template flow that --bootstrap creates:

appId: com.example.app
---
- launchApp
- assertVisible: ".*"

Line by line:

  • appId — the bundle identifier of the app to test (matches your config.json)
  • --- — separates config from the flow steps
  • launchApp — launches the app on the simulator/emulator
  • assertVisible: ".*" — asserts that any element is visible (basic "app starts" check)

Writing a custom flow

Add new .yaml files to .maestro/ for each user journey you want to test. A simple login flow might look like:

appId: com.example.myapp
---
- launchApp
- tapOn: "Log In"
- inputText:
    id: "email-field"
    text: "test@example.com"
- inputText:
    id: "password-field"
    text: "password123"
- tapOn: "Submit"
- assertVisible: "Welcome"

Maestro discovers all .yaml files in .maestro/ automatically — no registration step. Results appear as a "Maestro Flows" tile in the snagging checklist alongside Lighthouse and accessibility results.

💡
Let Claude write your flows. Instead of writing YAML by hand, tell Claude about your app's key user journeys: "Map out the main flows in my app and write Maestro test flows for each one in .maestro/". Claude can read your components, identify screens and navigation paths, and generate flows that match your actual element IDs and labels. This is much faster and less error-prone than writing flows manually — especially for apps with many screens.
ⓘ
Maestro troubleshooting. If Maestro flows fail to run, check that your simulator/emulator is booted and the app is installed. For timeout errors and other common issues, see Troubleshooting > Maestro.

Configuration

Your tests/config.json needs these fields for mobile projects:

{
  "projectType": "react-native",
  "appId": "com.example.myapp",
  "platform": "both"
}
  • projectType — one of "react-native", "expo", or "flutter". This is how DOE knows to use Maestro instead of Playwright.
  • appId — your app's bundle identifier, used by Maestro flows
  • platform — target platform: "ios", "android", or "both" (default)

Mobile Snagging Sections

When you run /snagging on a mobile project, three extra sections appear in the checklist:

  • Device Conditions — keyboard dismiss, background/foreground, force kill, orientation, low battery, interruptions
  • Accessibility Quick Check — VoiceOver/TalkBack, 44pt touch targets, Dynamic Type scaling, contrast, focus order
  • Network Resilience — airplane mode, slow connection, mid-operation disconnect, WiFi/cellular switching

These sections are designed to be completed on a real device. The responsive checklist HTML works on phones — tap pass/fail buttons directly while testing.

The Signposting System

One of the most useful things about DOE is that you never have to remember what to run next. Claude signals the state of the quality system at every step transition.

step completes → Claude runs the appropriate checks and reports pass / fail / skipped
all pass → Claude moves on without asking. Silence means green.
something fails → Claude explains what happened and proposes a fix. You approve or redirect.
manual check needed → Claude asks explicitly: "Can you check X in the browser?" No ambiguity.
feature complete → Claude runs /snagging automatically — automated test suite runs first, then the HTML checklist opens with results tiles and manual checks.

The rule is simple: if Claude doesn't ask, everything passed. You don't need to poll for status or remember to run checks yourself. The system tells you when it needs you.

✓
Tip: If you want to know the current quality status at any time, ask Claude directly — "What tests have run so far?" or "Is there anything failing?" Claude can give a full summary on demand.
