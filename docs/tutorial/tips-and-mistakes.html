Tips & Common Mistakes

What to do, what to avoid, and how to fix it when things go wrong.

Working with Claude Code through the DOE framework is smooth once you know the patterns. This page collects the habits that lead to productive sessions, the mistakes that cause friction, and troubleshooting steps for when something goes wrong.

Do This

These habits keep sessions productive and your project healthy.

Do

Be specific about what you want

Say "Build a search bar that filters recipes by name" instead of "add search." Specific prompts produce specific results. Vague prompts produce guesswork.

Do

Always /wrap before ending a session

Wrapping saves your session state to STATE.md so the next session picks up exactly where you left off. Skipping it means lost context.

Do

Read the stand-up card

When you run /stand-up, Claude reads your project state and tells you what needs attention — stale TODOs, uncommitted changes, audit warnings.

Do

Keep sessions focused — one task per session

A session that tries to do three things at once leads to confused context and tangled commits. One task, one session, one clean commit trail.

Do

Let contracts do the verification

Every task in todo.md has testable criteria. Run /agent-verify to confirm the work actually passes. Contracts don't lie.

Do

Use /scope when your idea is vague

If you're not sure exactly what you want to build, /scope runs a conversational session that turns a rough idea into a concrete brief with clear steps.

Do

Commit after every completed piece of work

Each completed task gets its own git commit. This makes rollback surgical — if something breaks, you revert one commit, not an entire session.

Do

Check STATE.md at the start of each session

Even before running /stand-up, a quick look at STATE.md reminds you where you left off and any blockers from last time.

Don't Do This

These are the most common mistakes. Each one has bitten someone before.

Don't

Paste API keys in the chat

Never share credentials in the conversation. Put them in your .env file, which is excluded from git. Execution scripts read from .env automatically.

Don't

Say "build me an app" with no context

Claude needs specifics. What does the app do? Who is it for? What data does it use? A single sentence of context produces dramatically better results.

Don't

Ignore audit warnings

When /audit flags something, it's catching a real problem — a stale claim, a misplaced file, a documentation gap. Ignoring it lets small issues compound.

Don't

Skip /wrap

Closing the terminal without wrapping means STATE.md won't reflect what happened. Your next stand-up will be working with stale information. It takes five seconds.

Don't

Try to do everything in one session

Long sessions accumulate context that makes Claude less precise. Break large features into steps, tackle one per session, and use /wrap + /stand-up between them.

Don't

Manually delete files to "fix" things

Tell Claude what to undo instead. Manual deletions leave orphaned references, break imports, and create state Claude doesn't know about. Let Claude handle file operations.

Don't

Bypass hooks with --no-verify

Git hooks exist to catch real problems — committed secrets, missing trailers, formatting issues. Disabling the safety net is almost never the right fix.

Don't

Run 4+ hours without wrapping

Long conversations accumulate context Claude struggles to track. Two focused 2-hour sessions with /wrap between them produce better results than one marathon session.

Managing Context

Understanding how Claude's working memory works — and how to manage it — is the single biggest factor in getting consistently good results across sessions.

Why context matters

Claude has a limited working memory called the context window. Everything in your conversation — your messages, Claude's responses, files it read, code it wrote — fills this up. When it gets full, Claude starts losing track of earlier details. This is normal, not a bug. Think of it like a desk that can only hold so many papers — eventually the oldest ones fall off the edge.

ⓘ
The goal isn't to avoid filling the context window. The goal is to fill it with the right things — current task, relevant files, recent decisions — not stale history from three hours ago.

When to /wrap

  • At the end of every work session (even a short one)
  • Before stepping away for more than a few minutes
  • When switching from one type of work to another

/wrap saves your progress to STATE.md and records session stats. Without it, the next stand-up won't know what happened. Think of it as pressing "save game" before turning off the console.

When to /clear

  • When the conversation feels "muddy" — Claude keeps referencing old context that's no longer relevant
  • When switching to a completely different task in the same project
  • After /wrap, if you want a clean slate for the next piece of work

/clear wipes the conversation but not your files. Your code, todo.md, STATE.md — all still there. You're resetting Claude's conversation memory, not emptying the filing cabinet.

The golden pattern

Session cycle 
$ /stand-up    # Read state, see what needs doing
$ ... work ...  # Focused on one task
$ /wrap        # Save progress to STATE.md
$ /clear       # Reset conversation memory
$ /stand-up    # Pick up the next task fresh

What Claude Reads

People often assume Claude remembers everything or knows nothing. The truth is in between — and understanding what Claude actually has access to will change how you work with it.

At the start of every conversation

Claude automatically reads these files before you say anything:

File What it contains
CLAUDE.md Project rules, operating instructions, directory structure. The "constitution" that shapes how Claude behaves in your project.
~/.claude/CLAUDE.md Your personal universal instructions. These apply to every project you work in.
ⓘ
Every time you open a new conversation, Claude reads the rule book. It knows how to work in your project before you say anything.

When you run /stand-up

Claude additionally reads:

File What it contains
STATE.md Where you left off, what's in progress, any blockers from last session.
tasks/todo.md The full task list with contracts and verification criteria.
learnings.md Project-specific lessons from past sessions.
ROADMAP.md The big picture of features planned, in progress, and done.

During work

  • Claude reads files on demand — when it needs to edit something, it reads it first
  • Claude does not have your whole codebase loaded at all times
  • If you want Claude to know about a specific file, mention it: "Check src/js/scoring.js"

What Claude does not automatically know

  • The contents of your code files (until it reads them)
  • What happened in previous conversations (unless saved to STATE.md or learnings.md)
  • What other people on your team are doing
  • Anything outside your project directory
ⓘ
STATE.md is a sticky note on the monitor — replaced every session. learnings.md is a notebook in the drawer — always growing, always available. Both matter, but for different reasons.

Keeping Your Kit Updated

The DOE creator releases improvements over time — new commands, bug fixes, smarter hooks. One command keeps you up to date.

/pull-doe

Pulls updates from the starter kit into your project. Like downloading an app update. Safe — won't overwrite your customisations.

When to use /pull-doe

  • The /stand-up card shows DOE KIT vX.Y.Z * pull — newer updates are available
  • You want new commands or bug fixes that have been released
  • Every week or two is a reasonable cadence

ⓘ

Troubleshooting

Things go wrong. That's normal — and DOE is designed to make recovery straightforward. Click any problem below for step-by-step fixes.

Claude seems confused or is going in circles

This usually means the conversation has accumulated too much context, or the task description is ambiguous.

Fix
  1. Run /clear to reset the conversation. This wipes the chat history but keeps all your files, git history, and project state intact. Nothing is lost.
  2. Start fresh with /stand-up.
  3. Describe what you want more specifically. Instead of "fix the layout," try "the header on the results page overlaps the chart on mobile — make the header stack above the chart on screens narrower than 768px."

If it keeps happening, the task might be too large. Break it into smaller, concrete steps.

Something broke and I don't know what changed

Every commit in git is a snapshot of your entire project. You can always see what changed and go back.

Fix
  1. Run git log --oneline to see your recent commits. Each line is a save point with a short description.
  2. Run git diff to see what's changed since the last commit — these are unsaved changes.
  3. If you need to undo the last change, ask Claude: "Revert the last commit." This undoes the most recent save point cleanly.
  4. If you need to go back further, tell Claude which commit you want to return to.

Don't panic. Every commit is a save point. Nothing in git is permanently lost unless you deliberately delete it.

Claude made a file in the wrong place

Claude follows the directory structure defined in CLAUDE.md, but occasionally drifts.

Fix
  1. Tell Claude to move it: "Move src/utils.js to execution/utils.py."
  2. Check the directory structure section in your CLAUDE.md — this is the reference for where files belong.
  3. Run /audit to check for structural issues across the project. The audit catches misplaced files, missing documentation, and other drift.
A contract check is failing

Contracts are the testable criteria that define "done" for each step. When one fails, something about the implementation doesn't match the specification.

Fix
  1. Run /agent-verify to see exactly which criteria are failing and why.
  2. Read the Verify: pattern carefully — it tells you precisely what's being checked. For example, file: src/chart.js contains renderChart means the file must contain the text renderChart.
  3. Common causes: a typo in a function or file name, a file wasn't saved before the check ran, or the build step wasn't run after editing source files.
  4. Claude will attempt up to 3 fixes automatically before asking for your help.
I closed the terminal without wrapping

This happens to everyone. Here's what it means:

  • Your code is safe. Everything that was committed to git is still there.
  • STATE.md won't reflect the last session. It still shows the state from your previous wrap, so the next stand-up will be slightly out of date.
Fix
  • Start a new session normally with /stand-up. Claude will read the git history and figure out where things are. It won't be as clean as a proper wrap, but it works.
  • If you want to be thorough, update STATE.md manually to note where you left off.

This isn't catastrophic, but building the /wrap habit avoids it entirely.

I don't understand what Claude is doing

It's important to understand what's happening in your project. Never let Claude keep building if you're lost.

Fix
  1. Run /sitrep for a plain-language status update — what's being worked on, what's been done recently, and what's coming next.
  2. Ask Claude directly: "Explain what you just did in simple terms." Claude will break down the last action without jargon.
  3. Check tasks/todo.md to see the current step and its contract. The contract tells you exactly what this step is supposed to achieve.
The audit is showing warnings

The /audit command checks your project's health — file structure, documentation freshness, claim accuracy, and more.

Fix
  1. Run /audit to see the full report.
  2. Understand the severity levels:
    • WARN means something needs attention but isn't broken — a document that hasn't been updated in a while, or a mild style inconsistency.
    • FAIL means something is actually wrong — a claim not backed by data, or a missing required file.
  3. Tell Claude to fix the issues: "Fix the audit warnings" or "Fix the audit failures." Claude can usually resolve them automatically.
I want to start over on a feature

Sometimes a feature isn't working out and you want a fresh start.

Fix
  1. Don't delete files manually — tell Claude what you want to undo.
  2. Git keeps every version of every file, so nothing is permanently lost. Claude can revert your project to the commit just before the feature started.
  3. Tell Claude: "Revert to before we started the [feature name] feature." Claude will find the right commit and take you back to that point.

Your previous work on the feature is still in git history if you ever want to reference it.

Emergency: something is really broken

If something has gone seriously wrong — corrupted files, a bad commit that broke everything, data that looks wrong — don't improvise.

Fix
  1. Read directives/break-glass.md — this file contains step-by-step recovery procedures for serious problems. It's written for exactly this situation.
  2. Don't run destructive commands like rm -rf or git reset --hard unless you understand exactly what they do. These commands can make things worse if used incorrectly.
  3. Ask Claude for help. Describe the error message or symptoms. Claude can diagnose most issues and will walk you through recovery.

The most important thing in an emergency is to stop and read before acting. Your git history is almost certainly intact, which means recovery is almost always possible.

Understanding Hooks

DOE uses two kinds of hooks: git hooks that run around git operations, and Claude Code hooks that run around Claude's tool use. Both are guardrails — they catch problems automatically so you don't have to remember to check.

What hooks are

Type Lives in What it does
Git hooks .githooks/ Scripts that run automatically before or after git operations. DOE ships a pre-commit hook (strips AI attribution from commits) and a commit-msg hook (validates message formatting).
PreToolUse hooks .claude/settings.json Rules that run before Claude uses a tool. Used to validate or block actions — for example, preventing destructive operations on protected files.
PostToolUse hooks .claude/settings.json Rules that run after Claude uses a tool. DOE uses these for tracking — updating session stats, logging tool use, recording progress to STATE.md.

How they're configured

  • Git hooks live in .githooks/ and are activated once with git config core.hooksPath .githooks
  • Claude Code hooks are defined in .claude/settings.json under the hooks key
  • The starter kit's setup.sh configures both automatically — no manual setup needed

What happens when a hook blocks you

When a hook fires, you'll see a message explaining what was checked and why it failed. Common examples:

  • A pre-commit hook might reject your commit if it detects an AI co-author trailer that shouldn't be there, or a file that shouldn't be committed (like .env)
  • A PreToolUse hook might block Claude from performing an action that conflicts with a guardrail
  • In both cases, the error message tells you exactly what was checked and what to fix

Here's what a typical pre-commit hook interaction looks like:

git commit 
$ git commit -m "Add feature"
[hook] Co-author trailer detected — stripping...
[hook] Commit message reformatted.
[main abc1234] Add feature
 2 files changed, 45 insertions(+)

How to debug a hook that blocks you

  1. Read the error message carefully — it tells you what the hook checked and why it failed
  2. Fix the underlying issue (don't bypass the hook)
  3. For git hooks: check .githooks/ to see the script source
  4. For Claude hooks: check .claude/settings.json to see the rule
  5. Retry the action after fixing
⚠
Never use --no-verify to bypass git hooks unless you understand exactly why the hook is blocking you. Hooks are guardrails — they catch real problems. Bypassing them silently introduces the issues they were designed to prevent.

When Things Go Wrong

Most problems in DOE have fast, well-worn recovery paths. Here are the four scenarios you're most likely to encounter.

A hook blocks your commit

The hook detected something it was designed to catch. This is the system working correctly.

Fix
  1. Read the error — it tells you exactly what's wrong.
  2. Fix the issue in your code or commit message.
  3. Try the commit again.
  4. If the hook itself appears buggy, check .githooks/ for git hooks or .claude/settings.json for Claude Code hooks.

Do not use --no-verify unless you fully understand the reason for the block.

Context runs out mid-task

Signs: Claude starts repeating itself, forgets earlier decisions, or quality drops noticeably. This means the conversation window is running low.

Fix
  1. Run /wrap immediately if you can — this saves your progress to STATE.md and learnings.md.
  2. If Claude is too confused to wrap properly, your git commits still have all the code. Nothing built is lost.
  3. Run /clear then /stand-up or /crack-on to start fresh with full context.
Prevention
  • Run /context periodically to check how full the window is.
  • Wrap proactively before reaching 50% — don't wait until quality degrades.
Claude seems confused or loops

Claude might start doing the same thing repeatedly, contradict earlier decisions, or seem stuck. This is usually a context window problem, not a code problem.

Fix
  1. Run /clear to reset the conversation completely. Your files and commits are untouched.
  2. Then run /crack-on to resume — it reads STATE.md and picks up where you left off.

Nothing in your project is affected by /clear. It only clears the conversation window.

Accidental bad commit

A commit went in that shouldn't have — wrong files, wrong message, or work that broke something.

Fix
  1. Don't panic — git tracks everything. Nothing is truly lost.
  2. The safest fix is git revert <commit-hash> — this creates a new commit that undoes the bad one without rewriting history.
  3. For serious problems (corrupted files, data damage), read directives/break-glass.md first. It has step-by-step procedures for exactly this situation.
  4. Never use git reset --hard or force-push without understanding the consequences.
✓
The most common recovery is the simplest: /wrap, /clear, /crack-on. This fixes 90% of quality issues by giving Claude a fresh context window with your full project state preserved in files.
