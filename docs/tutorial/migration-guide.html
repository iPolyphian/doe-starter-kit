Migration Guide

Upgrading your DOE project from any version to the latest.

Find Your Version

Before upgrading, you need to know which DOE version your project currently uses. There are several ways to check:

Terminal 
# Method 1: Check STATE.md for the version line
$ grep "DOE Starter Kit" STATE.md
DOE Starter Kit: v1.42.0

# Method 2: Check the version file (if it exists)
$ cat .doe-kit-version
1.42.0

# Method 3: Check the latest tag in the starter kit repo
$ cd ~/doe-starter-kit && git log --oneline | head -5
a1b2c3d Save session 206 wrap data
d4e5f6g Bump to v1.54.2
...

# Method 4: If none of the above work, you're on a pre-v1.20 version

Once you know your version, find it in the table below to understand which eras of changes you need to catch up on, then follow the relevant migration paths.

Version History at a Glance

DOE has evolved through 10 distinct eras, each adding a layer of capability. This table shows what changed and when.

Migration Paths

Find your current version in the list below and follow the instructions for each breakpoint between your version and the latest. If you're on v1.45, for example, you'd follow the v1.47, v1.49, v1.51, and v1.52 paths in order.

Upgrading to v1.49 (Context-First Architecture) High Impact

What changed: CLAUDE.md was rewritten from a ~113-line monolith to a ~55-line thin router. Numbered rules (Rule 1–9) were replaced with a trigger table pointing to phase directives. Six new directive files were added to handle the rules that previously lived inline.

What you need to do:

1

Back up your current CLAUDE.md

Run cp CLAUDE.md CLAUDE.md.backup so you have a reference of your customisations.

2

Read the new CLAUDE.md template

Open the kit's CLAUDE.md and compare the structure. Note how rules have been replaced with a trigger table and how the directory structure section is preserved.

3

Copy the new thin router structure

Replace your CLAUDE.md with the new template. Keep your project's Directory Structure section and any custom triggers you've added — merge them into the new format.

4

Copy 6 phase directives

Copy these files into your directives/ folder: planning-rules.md, building-rules.md, delivery-rules.md, context-management.md, self-annealing.md, framework-evolution.md.

5

Update references to numbered rules

Search tasks/todo.md and any plan files for references to "Rule N" — replace them with pointers to the relevant directive file instead.

6

Install DAG executor

Copy dispatch_dag.py to ~/.claude/scripts/. This enables parallel step dispatch for multi-agent workflows.

7

Install custom agents

Copy the .claude/agents/ directory from the kit. This includes Finder, Adversarial, Referee, and ReadOnly agent definitions.

8

Install pre-push hook

Copy .githooks/pre-push from the kit and run chmod +x .githooks/pre-push. Ensure hooks are active with git config core.hooksPath .githooks.

⚠
What breaks if you skip this: Phase directives won't load. Commands that reference directives (stand-up, crack-on, wrap) may show warnings. DAG parallel dispatch will be unavailable.

Upgrading to v1.51 (Verification & Security) Medium Impact

What changed: Executable Verify patterns were formalised (run:, file: exists, file: contains, html: has). A review gate now blocks PR creation without Finder review. Step-marking enforcement hooks were added. Main branch protection was introduced.

What you need to do:

1

Update contracts in todo.md

Ensure all [auto] criteria use one of the 4 Verify patterns: run:, file: exists, file: contains, or html: has. Old-style free-text criteria won't be validated automatically.

2

Install enforcement hooks

Copy these files to .claude/hooks/: enforce_review_gate.py, record_review_result.py, persist_review_findings.py.

3

Update pre-commit hook

Update .githooks/pre-commit with the step-marking and main-branch protection sections from the kit's version.

4

Update settings.json

Update .claude/settings.json with the new hook registrations for PreToolUse and PostToolUse events.

⚠
What breaks if you skip this: Old-style contracts ("Verify: check that X works") won't be validated by /agent-verify. PR creation may fail if the review gate hook is partially installed.

Upgrading to v1.52 (Init Wizard & Manifest) Medium Impact

What changed: manifest.json was introduced to track which files belong to each capability layer. The init wizard replaced setup.sh for new projects. Settings.json hooks were wired up. Missing files were added (.claude/plans/, .claude/stats.json, ROADMAP.md).

What you need to do:

1

Copy manifest.json

Copy manifest.json from the kit to your project root. This file maps every DOE file to its capability layer.

2

Verify settings.json hooks

Open .claude/settings.json and confirm it has PreToolUse and PostToolUse hooks wired. Compare with the kit's version.

3

Create missing files

Create any files that don't exist yet: .claude/stats.json, ROADMAP.md, tasks/archive.md. These can be empty initially.

4

Copy agents directory

If your .claude/agents/ directory is missing, copy it from the kit. This provides the agent definitions used by /review and multi-agent workflows.

⚠
What breaks if you skip this: /doe-health may report missing files. Some hooks may not fire. New commands expecting manifest.json will warn.

Upgrading to v1.47 (Agent Discipline) Low Impact

What changed: Rationalisation tables, serial dispatch protocol, TDD directive, and Chrome verification were added. All changes are additive — no structural modifications to existing files.

What you need to do:

1

Copy new directives

Copy these to directives/: rationalisation-tables.md, serial-dispatch-protocol.md, subagent-protocol.md, best-practices/tdd-and-debugging.md, chrome-verification.md.

2

Add triggers to CLAUDE.md

Add entries to your CLAUDE.md trigger table for the new directives: rationalisation tables, serial dispatch, TDD, and Chrome verification.

3

Copy CI pipeline

Copy .github/workflows/doe-ci.yml and .github/workflows/auto-rebase.yml from the kit to enable automated checks on push and automatic rebase for queued PRs.

ⓘ
Nothing breaks if you skip this. These are purely additive features. You just won't have access to rationalisation tables, serial dispatch, or the CI pipeline.

CLAUDE.md Migration in Detail

The CLAUDE.md restructuring in v1.49 is the most involved upgrade. The file went from a monolithic rule set to a thin router. Here's what the before and after look like:

Before (pre-v1.49)

# Rules
Rule 1: Plan before building (... 20 lines of detail ...)
Rule 2: Ask, don't assume
Rule 3: One task, one session
...
Rule 9: Check before spending

# Guardrails
IMPORTANT: Never commit .env files
IMPORTANT: Run builds after editing src/
...

# Progressive Disclosure
- When planning -> planning-related rules
- When building -> building-related rules

After (v1.49+)

## Architecture: DOE
Directive -> Orchestration -> Execution. Non-negotiable.

## Core Behaviour
1. Plan before building. -> directives/planning-rules.md
2. Ask, don't assume.
3. Check before spending.
4. Verify before delivering. -> directives/delivery-rules.md
5. Explain simply.
6. One task, one session. -> directives/building-rules.md
7. Shared-file awareness. -> directives/context-management.md

## Triggers
- Planning/scoping -> directives/planning-rules.md
- Building/coding -> directives/building-rules.md
- Delivering -> directives/delivery-rules.md
- Parallel work -> directives/context-management.md
- Self-annealing -> directives/self-annealing.md
- Platform evolution -> directives/framework-evolution.md

The key difference: rules are now one-liners with pointers, not paragraphs. The detail lives in directive files that only get loaded when relevant.

Preserving Your Customisations

ⓘ
Your customisations survive the migration. Here's where each type goes in the new structure:
  • Custom triggers: Add them to the Triggers table in the new format (- When X happens -> directives/your-file.md).
  • Project-specific directory entries: Keep them in the Directory Structure section — this section is unchanged.
  • Custom rules: Move them to a project-specific directive file (e.g. directives/project-rules.md) and add a trigger for when they apply.

Fresh Start vs Surgical Upgrade

There are two approaches to upgrading. Which one to use depends on how far behind you are and how much you've customised.

Fresh Start

Start from the latest template and re-add your customisations.

Surgical Upgrade

Update your existing files section by section.

Choose Fresh Start when:

  • Your CLAUDE.md is heavily customised and you're not sure what you changed vs what came from the kit
  • You're more than 2 eras behind (e.g. upgrading from v1.30 or earlier)
  • You'd rather start clean and selectively re-add your customisations

How: Copy the kit's CLAUDE.md template into your project, then add your custom triggers, directory entries, and project-specific rules back in. Use your CLAUDE.md.backup as reference.

Choose Surgical Upgrade when:

  • You're only 1–2 versions behind
  • You have few customisations or they're well-organised
  • You want to understand exactly what changed and why

How: Follow the per-breakpoint migration paths above, comparing each section of your files with the kit's version and applying changes incrementally.

Post-Migration Verification

After upgrading, run these checks to confirm everything is wired up correctly:

1

Run /doe-health

All methodology tests should pass. Some WARNs are expected on fresh upgrades (session_start_discipline, claude_md_quality) — these reflect missing session history, not broken configuration. Errors indicate missing files that need to be copied from the kit.

2

Run /stand-up

The kick-off card should render without errors. The DOE KIT line should show your current version number. If the card fails to render, check that STATE.md exists and has the expected structure.

3

Try /crack-on on a small task

Verify directive loading works — you should see Claude reading the relevant directive file (e.g. directives/building-rules.md) at the start of the session. If it doesn't load any directives, check the trigger table in CLAUDE.md.

4

Check hooks are active

Run git config core.hooksPath — it should show .githooks. Try a test commit to verify pre-commit hooks fire. If hooks aren't active, run git config core.hooksPath .githooks.

5

Investigate any failures

If anything fails, check the specific breakpoint section above for what might be missing. Most issues come from a file that wasn't copied or a hook that wasn't registered.

ⓘ
Stuck? Run /report-doe-bug to file an issue with your environment details automatically collected. This captures your version, installed files, hook status, and recent errors — everything needed to diagnose the problem.
