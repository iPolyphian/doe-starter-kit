Getting Started

Everything you need to install DOE and Claude Code — even if you've never used a terminal before.

Prerequisites

Before you can use DOE, you need three things set up on your machine. Each card below explains what it is, why you need it, and how to get it.

VSCode

Visual Studio Code is a free code editor. You'll use it to see your project files and run Claude Code. Download from code.visualstudio.com. It has a built-in terminal so you don't need a separate app.

Git

Version history for code, like Google Docs version history but for files. Every time you finish work, you "commit" it — like saving your game. Download from git-scm.com. Mac users: it may already be installed.

Claude Code

The AI coding assistant that powers DOE. You need a Claude Pro or Max subscription. Install via the VSCode marketplace or the Claude Code docs. No API key needed — authentication is automatic.

ⓘ
Opening your project: Open VSCode, then either drag your project folder into the window or use File > Open Folder. The integrated terminal (Ctrl+` or Cmd+` on Mac) is where you'll run Claude Code. Alternatively, on Mac press Cmd+Space, type "Terminal", hit Enter. On Windows, search for "Windows Terminal" in the Start menu.
Understanding Git (optional)

Git came up in the prerequisites. If you've never used version control before, here's a quick glossary. Claude handles all of this for you — these terms just help you follow along.

Repository
A project folder that remembers every change you've ever made. Your DOE project is a repository.
Commits
Save points. Each commit snapshots your project at a moment in time. If something breaks, you can go back.
Push / Pull
Cloud sync. Push sends your save points to GitHub (backup). Pull downloads the latest version.
Branches
Parallel versions. Like editing a copy while the original stays safe.
Why DOE commits after every task
DOE automatically creates save points after each completed task. This means you always have a clear history of what changed and can undo any single change without losing everything else.

You don't need to memorise these — Claude handles git for you. But knowing the words helps you understand what's happening when Claude says "committed" or "pushed".

Choose Your Setup

There are two paths to get DOE running. Both give you the same result — a project folder with all the configuration, commands, and scripts that make DOE work.

  1. 1

    Clone the repository

    Open your terminal and run:

    git clone https://github.com/iPolyphian/doe-starter-kit.git my-project
cd my-project

    Replace my-project with whatever you want your folder to be called. You can also download the ZIP from GitHub if you prefer not to use Git for this step.

  2. 2

    Run the setup script

    bash setup.sh

    This installs slash commands (/stand-up, /wrap, etc.), sets up Git hooks, creates the directory structure DOE expects, and initialises Git if needed. It will ask if you're a DOE contributor — just press Enter to select the default (no). This is the right choice for most users.

  3. 3

    Verify it works

    claude

    Claude Code's interface should appear. Type /commands to confirm the DOE commands are available. You should see /stand-up, /wrap, and /sitrep in the list.

If you already have a project with code in it, the easiest approach is to clone the starter kit and copy the DOE files into your project.

  1. 1

    Clone or download the starter kit

    git clone https://github.com/iPolyphian/doe-starter-kit.git doe-kit

    This gives you a temporary copy of the kit. You can also download the ZIP from GitHub.

  2. 2

    Copy the DOE files into your project

    The kit includes everything you need. Copy these files and folders from doe-kit/ into your project root:

    • CLAUDE.md — the project instruction file
    • directives/ — SOP documents for specific tasks
    • execution/ — deterministic Python scripts
    • tasks/todo.md — task tracker with contracts
    • STATE.md — session memory
    • learnings.md — project learnings log
    • .claude/commands/ — slash commands
    • .githooks/ — Git hooks for quality checks
  3. 3

    Run the setup script

    cd ~/your-existing-project
bash setup.sh

    This configures Git hooks and verifies the directory structure is correct. When it asks about DOE contributor, just press Enter for the default (consumer).

  4. 4

    Verify it works

    claude

    Open your project in the terminal and run claude, then type /commands to verify the DOE commands are available.

Configure Your Project

CLAUDE.md — The Project Instruction File

CLAUDE.md is the single most important file in DOE. It's the instruction manual that Claude reads at the start of every conversation. It controls how Claude behaves: what rules it follows, what it's allowed to do, and where it puts files.

The starter kit gives you a solid default. Before your first real session, customise these parts:

  • Directory Structure — must match your actual project folders. Claude uses this to know where to create new files.
  • Progressive Disclosure triggers — tell Claude "when you're about to do X, read file Y first." Add 2–3 for your project's key areas.
  • Project-specific guardrails — rules that protect your project (e.g., "never delete user data without confirmation").

CLAUDE.md example

Here's a condensed version of what a real CLAUDE.md looks like. The starter kit includes the full version — this shows the key sections:

# Project Configuration

## Who We Are
The human defines intent, constraints, and verification criteria.
Claude recommends technical approach, explains trade-offs simply,
then implements. The human steers — Claude builds.

## Operating Rules
1. Plan before building. Check tasks/todo.md at session start.
2. Ask, don't assume. If a requirement is ambiguous, ask.
3. Verify before delivering. Run the script, test the output.
4. Commit after every completed task. Never batch multiple tasks.

## Directory Structure
directives/         # SOPs for specific tasks
execution/          # Deterministic Python scripts
tasks/todo.md       # Task tracker with contracts
STATE.md            # Session memory
learnings.md        # Project learnings log
.env                # API keys (NEVER commit)

## Progressive Disclosure
- Importing external data → check learnings.md
- Building a UI component → check directives/ui-patterns.md
- Version bump or release → check tasks/todo.md
ⓘ
Start simple. You can always add more rules as you learn what works for your project. The starter kit's CLAUDE.md has the full set of rules — read through it and remove anything that doesn't apply to you.
ⓘ
The 200-line rule: Keep CLAUDE.md under ~200 lines. Claude reads the whole thing at the start of every conversation, and every line uses context window space. If something needs more detail, put it in a directive file and add a trigger.

Directory Structure

After setup, your project has this structure. You don't need to memorise it — Claude knows where everything goes.

my-project/
  CLAUDE.md          # Project instruction file (Claude reads this first)
  STATE.md           # Session memory (updated by /wrap)
  learnings.md       # Project learnings log (grows over time)
  directives/        # SOP documents for specific tasks
  execution/         # Deterministic Python scripts
  tasks/
    todo.md          # Task tracker with contracts
  .env               # API keys and credentials (NEVER commit)
  .claude/           # Hooks, settings, plans
  .githooks/         # Git hooks for quality checks

API Keys and .env

If your project uses external APIs, store the keys in a .env file in the project root. This file is automatically excluded from Git so your secrets stay safe. Claude Code reads .env automatically when running execution scripts.
