Context & Sessions

The most important operational concept in AI-assisted development — and the one that catches everyone out.

What is the Context Window

Think of the context window as Claude's desk. Everything Claude can see right now — your messages, its responses, files it has read, tool outputs, system instructions — is spread across this desk. Claude can only work with what's on the desk. Anything that falls off the edge is gone.

The current context window is approximately 1 million tokens (roughly 750,000 words). That sounds enormous — and it is. But it fills up faster than you'd expect. A single large source file might consume thousands of tokens just by being read. Every message back and forth adds more. Tool outputs from running scripts pile up. System prompts, memory files, and CLAUDE.md all take space before you've even started working.

Bigger doesn't mean infinite. Even with a million tokens, quality starts to degrade well before the window is full. Claude's attention is strongest on recent messages and weakest on content from the start of a long conversation. The practical limit is significantly lower than the theoretical maximum.

In Plain English

Claude has a limited amount of text it can "see" at once. Everything in the conversation uses some of that space. When it fills up, Claude starts forgetting things and making mistakes. DOE's session discipline is designed to prevent this from ever becoming a problem.

The Danger Zones

Context usage isn't a smooth gradient from "fine" to "broken". There are distinct thresholds where behaviour changes noticeably.

Context Window Usage
0 – 50%
50 – 70%
70 – 85%
85%+
0% 50% 70% 85% 100%
  • 0 – 50% (green): Full performance. Claude tracks all decisions, remembers earlier context, works accurately. This is where you want to be.
  • 50 – 70% (yellow): Claude starts to lose track of earlier decisions. You may notice it asking about something you already discussed, or slightly contradicting an earlier plan. Quality is noticeably lower than a fresh session.
  • 70 – 85% (orange): The danger zone. Claude may forget what it agreed to, re-read files it already looked at, or contradict itself. Errors become more frequent. This is where you should wrap up and start fresh.
  • 85%+ (red): Critical. Multiple compactions have likely occurred. Claude is working from summaries of summaries. You will lose work at this level — decisions get forgotten, code gets overwritten, entire sections of progress can vanish.
⚠
The 50% rule: Wrap proactively when you hit ~50% context usage. It sounds early, but the quality difference between a fresh session and a 60% session is significant. Your future self will thank you.

Why One Task Per Session

Context is a finite resource. Every message, file read, and tool output consumes some of it. If you spend context on unrelated topics — discussing deployment, then switching to a UI bug, then asking about database schemas — you're diluting Claude's focus and burning through your window faster.

Tightly grouped tasks keep Claude accurate. When every message in the conversation is about the same feature, Claude holds all the relevant details in sharp focus. There's no noise from unrelated topics competing for attention.

If a session drifts to a different topic, the right move is to wrap the current session and start fresh. The few minutes it takes to wrap and restart are nothing compared to the errors you'll get from a cluttered, unfocused context.

DOE's session discipline — /stand-up to begin, work on one thing, /wrap to save progress — is designed around this constraint. It's not bureaucracy; it's context hygiene.

ⓘ
Rule of thumb: If you find yourself saying "while we're at it, can you also..." — that's a signal to finish the current task, wrap, and handle the new thing in a fresh session.

How Compaction Works

When the context window fills up, Claude Code automatically compacts the conversation. Compaction summarises earlier messages to free up space, keeping the most recent context intact while condensing older content into shorter summaries.

The first compaction is tolerable. Some detail is lost, but the main thread of the conversation is preserved. You might lose the exact wording of an earlier decision, but the gist survives.

The problem is multiple compactions. Each compaction summarises what came before — including previous summaries. After two or three rounds, you're working from a summary of a summary of a summary. Critical details are gone. Decisions you made early in the session have been compressed into vague approximations.

⚠
Multiple compactions will cost you real work. Decisions get forgotten. Code gets overwritten. Claude may redo work it already completed because it can't remember having done it. This is why DOE wraps sessions proactively — it's cheaper to spend 2 minutes wrapping than to lose 30 minutes of progress.

Compaction is a safety net, not a feature. You should never plan to rely on it. If your workflow regularly triggers compaction, your sessions are too long or too unfocused.

Here's what the numbers mean:

  • System prompt: CLAUDE.md, project rules, and Claude Code's built-in instructions. This is fixed overhead — it's there before you say anything.
  • Tools: The definitions of all available tools (Bash, Read, Edit, etc.). Also fixed overhead.
  • Memory files: STATE.md, learnings.md, and any other memory files Claude loads at session start.
  • Skills: Custom slash commands loaded from .claude/commands/.
  • Messages: Everything you and Claude have said, plus file contents read and tool outputs. This is the part that grows as you work.
  • Free space: What's left. When this gets low, you're in trouble.
ⓘ
When to worry: If the percentage is above 50%, consider wrapping soon. If it's above 70%, wrap now. Run /context periodically — especially during long or complex tasks.

/wrap — Your Safety Valve

The /wrap command is how you save your session's progress before clearing the conversation. It's the single most important habit in DOE.

What /wrap saves

  • STATE.md — Your current position: what feature you're working on, what step you're at, any blockers, and what to do next. This is the short-term memory that lets the next session pick up where you left off.
  • learnings.md — Anything discovered during the session: patterns, gotchas, things that failed and why. This is the long-term memory that makes the project smarter over time.
  • Session stats — Duration, commits made, files changed, and other metrics. Useful for tracking velocity and patterns.

What /wrap does NOT save

  • The conversation itself. Once you /clear, the messages are gone. Claude won't remember what you discussed. This is by design — the conversation is the most expensive part of context, and starting fresh is the whole point.
  • Uncommitted code changes. Wrap saves metadata, not code. If you have uncommitted work, commit it before wrapping. (DOE commits after every completed task, so this is rarely an issue.)

The golden pattern

This is the session lifecycle that keeps context clean and progress safe:

Work on a focused task. /wrap to save progress to STATE.md and learnings.md. /clear to reset the conversation. /stand-up (or /crack-on) to begin the next session with a fresh context and full awareness of where you left off.

What To Do When Context Runs Out

Even with good habits, you'll occasionally find yourself in a high-context situation. Here's how to handle it.

If quality is dropping mid-task

You'll notice the signs: Claude re-reads files it already looked at, asks about decisions you already made, or produces code that contradicts earlier work. When this happens, run /wrap immediately, then /clear. Don't try to push through — the errors will compound.

If Claude seems confused or loops

Sometimes Claude gets stuck in a loop — trying the same fix repeatedly, or going back and forth between two approaches. This is almost always a context problem. Wrap, clear, and start fresh. A clean context often resolves the issue instantly because Claude can approach the problem without the noise of previous failed attempts.

If you lost context unexpectedly

Don't panic. Your progress is safe in multiple places:

  • Git commits have all your code changes. DOE commits after every completed task, so your work is captured.
  • STATE.md has your position — what you were working on and what's next (assuming you wrapped before losing context).
  • learnings.md has anything discovered during the session.
  • tasks/todo.md has the full task breakdown with contract checkboxes showing what's been completed.

Nothing built is ever lost. Only the conversation is gone — and that's the disposable part. Start a new session with /stand-up and Claude will rebuild its understanding from the files.

⚠
Prevention beats recovery. Run /context periodically. Wrap proactively at ~50%. The habit of wrapping early saves more work than any other DOE practice.

Quick reference

# Feeling like context is getting heavy?
/context              # Check usage

# Above 50%? Wrap and start fresh:
/wrap                 # Save progress
/clear                # Reset conversation
/stand-up             # Fresh briefing

# Need to immediately continue building?
/crack-on             # Skip briefing, resume work
