Your First Session

Walk through building something with DOE from start to finish — from opening the terminal to wrapping up with all your progress saved.

The Session Rhythm

Every DOE session follows the same three-beat rhythm. Once you've done it a few times, it becomes second nature.

Stand-up

Get oriented. See where the project stands and what's next.

Describe & Build

Tell Claude what you want in plain English. Review the plan, then build.

Wrap

Save your position and session memory so next time picks up seamlessly.

Let's walk through each beat using a real example: building a recipe book app.

Starting Up

Open your terminal, navigate to your project folder, and start Claude Code. If you're using VSCode, open your project folder first (File > Open Folder or drag the folder into the window), then open the integrated terminal with Ctrl+` (or Cmd+` on Mac).

cd ~/my-project
claude

Once Claude Code's prompt appears, run the stand-up command:

/stand-up

On your first session, most of this is empty — and that's fine. The card becomes more useful as sessions accumulate. The stand-up shows:

  • Feature — what you're currently building
  • Progress — how far through the current work you are
  • Plan — the steps, with checkmarks for completed ones
  • Blockers — anything preventing progress
  • Last session — a brief summary of what happened last time

Building

Now describe what you want to build. Just type in plain English — be specific about the end result, not the technical details:

I want to build a recipe book app where I can add recipes with a title,
ingredients list, and method. I want to search recipes by name or
ingredient, and mark my favourites so I can find them quickly.

Claude follows a structured process:

  1. Reads CLAUDE.md — your project's instruction manual that tells Claude the rules
  2. Creates a plan — breaks your request into steps with clear success criteria (contracts), written to tasks/todo.md
  3. Asks for approval — shows you the plan before building anything

This is your chance to adjust scope, reorder priorities, or ask questions. Nothing gets built until you say go.

Once you approve, Claude works through the plan step by step. After each step, it verifies the work against the contracts and commits the result as a Git save point. Each commit is like a save point in a game — clearly labelled so you know exactly what it contains. If step 5 breaks something, you can go back to the state after step 4 without losing anything.

You can talk to Claude throughout this process. Ask questions, change direction, or tell it to skip something. It's a conversation, not a one-way instruction.

The wrap command:

  • Saves your current position — updates STATE.md so the next session knows exactly where you left off
  • Records session stats — how long you worked, how many commits were made
  • Summarises the session — creates a brief record of what happened
ⓘ
Why this matters: Claude Code doesn't automatically remember previous sessions. Without the wrap-up, your next session would start cold. When you come back — whether in an hour, a day, or a month — your next /stand-up will show everything the wrap saved.

What Just Happened?

In this first session, here's what DOE did behind the scenes:

  • Checked STATE.md — read the project's current position (empty on first session)
  • Followed the plan — worked through steps in order, never jumping ahead
  • Verified contracts — ran automated checks after each step to prove the work actually functions
  • Committed after each step — created atomic save points so any step can be reverted independently
  • Updated session stats — tracked duration, commits, and progress
  • Saved state — wrote everything to STATE.md so the next session picks up seamlessly

This is the rhythm every session follows: stand-up, build, wrap. The more sessions you run, the richer your project's memory becomes — and the more effectively Claude can build on previous work.

Sessions and State

Claude Code doesn't have built-in memory between sessions. When you close it, the conversation is gone. DOE solves this with STATE.md — a file that captures your project's current position so the next session can pick up exactly where you left off.

STATE.md tracks:

  • Current feature — what you're building right now
  • Progress — which steps are done and which are next
  • Blockers — anything preventing progress
  • Last session summary — a brief record of what happened
  • Decisions made — choices that affect future work

Here's what STATE.md looks like after a few sessions:

## Current Position
Feature: Recipe Book App [APP] v0.1.x
Step: 4 of 5 (Favourites system)
Blocker: None

## Last Session (Session 2)
Duration: 35 min | Commits: 3
Completed search functionality and fixed edge case
where empty searches returned no results instead of
showing all recipes. Next: implement favourites.

When you run /stand-up next time, Claude reads this file first. Within seconds it knows what you're building, where you left off, and what to do next — even if weeks have passed.
