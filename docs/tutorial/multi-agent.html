Multi-Agent Workflows

Run parallel Claude sessions to build faster.

What Are Waves

In Plain English

A wave is like having multiple Claude assistants working simultaneously — each in their own terminal, each on a different task. A coordinator manages the work from a separate terminal. When everyone’s done, the coordinator merges the results back together.

Waves split independent tasks across terminals. Each agent gets an isolated copy of the code (a “worktree”), works on its task without affecting the others, and the coordinator brings all the changes back together when everyone finishes. Think of it as a team of builders working on different rooms of the same house — they don’t need to share tools or get in each other’s way.

The key requirement is independence. Each task in a wave must touch different files. If two tasks need to edit the same file, they can’t run in parallel — one must finish before the other starts.

When to Use Waves vs Single Session

⚙

Use a Single Session When

Tasks depend on each other. You’re exploring or designing. There are fewer than 3 tasks. Tasks share the same files.

⚡

Use Waves When

You have 3+ independent tasks. Tasks touch different files. You want to finish faster. Each task has clear contracts.

ⓘ
Rule of thumb: If you can draw clear lines between which files each task touches and none of them overlap, waves will speed things up. If you’re not sure, start with a single session — you can always parallelise later once the plan is clearer.

How /agent-launch Works

From your coordinator terminal, run /agent-launch. Claude analyses your todo.md to find tasks that can run in parallel, then shows you what it found.

Coordinator Terminal 
$ claude
> /agent-launch

Wave Setup
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Tasks eligible for parallel work:
  1. Step 3: Add search component    [owns: src/js/search.js, src/styles/search.css]
  2. Step 4: Add filter sidebar       [owns: src/js/filters.js, src/styles/filters.css]
  3. Step 5: Add sort controls        [owns: src/js/sort.js]

Shared files (coordinator only): todo.md, CLAUDE.md, learnings.md

Launch 3 agents? [y/n]

Each agent gets its own terminal and its own worktree — an isolated copy of the codebase where it can make changes without interfering with other agents. The coordinator terminal stays open for monitoring progress.

The [owns: ...] tags in each task tell the system which files that agent is allowed to edit. This is the contract that prevents conflicts — no two agents can own the same file.

Worked Example: Building 3 Independent UI Components

You’re building an app and your todo.md has three steps ready to build, each touching different files. Here’s how a wave plays out from start to finish.

  1. 1

    Identify parallel tasks in todo.md

    Three steps are in the Queue, each with an [owns: ...] block listing different files. No overlap means they can run simultaneously.

  2. 2

    Open coordinator terminal, run /agent-launch

    Claude shows the three eligible tasks and asks for confirmation. You say yes. Three new terminal windows open automatically.

  3. 3

    Agents work independently

    Each agent reads its task’s contract from todo.md, builds the feature, runs the [auto] verification criteria, and commits when done. They don’t see each other’s work.

  4. 4

    Monitor with /agent-status

    From the coordinator terminal, check progress at any time. You’ll see which agents are done, which are still building, and whether any issues have come up.

  5. 5

    Agents complete — coordinator merges

    When all agents are done, run --complete to verify each agent’s work. Each agent creates a pull request from their feature branch, then --merge brings all changes back into the main branch. Three features built in the time it takes to build one.

The Merge Process

When agents finish their work, the coordinator brings everything together in two steps.

Step 1: Verify with --complete

Run --complete from the coordinator terminal. This re-runs each agent’s contract criteria to confirm everything still passes. If an agent’s work fails verification, you’ll see which criteria failed and can fix them before merging.

Step 2: Merge with --merge

Once all agents pass verification, run --merge to bring their changes into the main branch. The coordinator then updates shared files — marking steps complete in todo.md, recording any learnings, and updating STATE.md.

Coordinator Terminal 
> --complete

Verifying agent work...
  Agent 1 (search):   4/4 criteria pass
  Agent 2 (filters):  5/5 criteria pass
  Agent 3 (sort):     3/3 criteria pass

All agents verified. Ready to merge.

> --merge

Merging wave-20260315-1430...
  Merging agent-1 (search)...     done
  Merging agent-2 (filters)...    done
  Merging agent-3 (sort)...       done

Cleaning up worktrees...          done
Updating todo.md...               done

Wave complete. 3 steps merged successfully.

Shared file rules

During a wave, agents must not edit shared files. These include:

  • todo.md — task tracking (coordinator updates after merge)
  • CLAUDE.md — project instructions
  • learnings.md — institutional memory
  • STATE.md — session state

The coordinator is the only one who touches these files, and only after the merge is complete. This rule prevents merge conflicts and keeps the project’s single source of truth intact.

Common Pitfalls

⚠
Editing shared files from an agent. If an agent edits todo.md, learnings.md, or other shared files, the merge will conflict. The coordinator handles all shared file updates after the wave completes.
⚠
Tasks that depend on each other. If Step 4 needs the output of Step 3 (imports a module it creates, reads a file it generates), they cannot run in parallel. Finish Step 3 first, then launch Step 4. Dependencies must be sequential.
⚠
Too many agents at once. Each agent uses its own terminal and memory. Running 6+ agents on a laptop will slow everything down. The sweet spot is 3–4 agents for most machines.
⚠
Forgetting to verify before merge. Always run --complete before --merge. Merging unverified work can introduce bugs that are harder to track down after the worktrees are cleaned up.
