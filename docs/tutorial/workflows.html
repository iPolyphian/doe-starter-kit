Workflows

From idea to shipped feature — three starting points, one consistent pipeline.

Not every project starts the same way. Sometimes you have no idea what you’re building. Sometimes you have bullet points. Sometimes you have a full spec. DOE handles all three — the workflow adjusts to match your starting point, then converges on the same build pipeline once the plan is ready.

Choose Your Starting Point

Pick the path that matches where you are right now. All three paths lead to the same place: a plan in .claude/plans/ with contracted steps in todo.md, ready to build.

Path A: “I have no idea what I’m building”

You know you want to build something, but you haven’t pinned down what. Maybe you have a domain (“something with recipes”) or a problem (“I keep losing track of things”) but no shape to it yet.

  1. 1

    Run /scope

    Claude asks you questions to pin down what you actually want — who it’s for, what the output looks like, what counts as “done.”

  2. 2

    Scoping session produces a brief

    The conversation outputs a short document capturing what you’re building, the constraints, and the definition of done.

  3. 3

    Brief becomes a ROADMAP.md entry

    The scoped feature gets added to your roadmap so it’s visible in the big picture alongside everything else.

  4. 4

    Claude writes the plan

    A plan file in .claude/plans/ with sequenced steps, each with contracts defining what “done” means.

  5. 5

    Build

    Tell Claude what to build in plain English, or say “go” to start the next planned step. Claude builds, verifies each contract, and commits. Use /crack-on if you get interrupted and need to resume later.

ⓘ
When to use Path A: When you have a domain or a problem but no shape to the solution. The scoping conversation costs five minutes and saves hours of rework by clarifying requirements before any code is written.

Path B: “I have a rough idea”

You’ve thought about it enough to have bullet points. You know the key features, maybe the tech stack, and roughly what the end result looks like. You just haven’t structured it into a buildable plan.

  1. 1

    Give Claude your bullet points

    Describe what you want in plain English. List the features, the data sources, who it’s for. No special format needed.

  2. 2

    Claude creates the plan

    A plan file with sequenced steps, technical approach, and the right model/effort level for each step.

  3. 3

    Steps added to todo.md with contracts

    Each step gets testable criteria — [auto] for things Claude verifies itself, [manual] for things that need your eyes.

  4. 4

    Start building

    Tell Claude to build the next step, or just say “go.” Each step is built, verified, and committed independently. If you get interrupted later, /crack-on resumes where you left off.

ⓘ
When to use Path B: When you already know the key features and the general shape. You don’t need a scoping session — Claude has enough to create a plan directly from your description.

Path C: “I have a full spec”

You’ve written a detailed specification. Named technologies, specific features, clear requirements, maybe even a phased roadmap. You want Claude to execute it faithfully, not reinterpret it.

  1. 1

    Import spec as a directive

    Save your spec to directives/. Claude treats this as the authority — it follows the spec, not its own preferences.

  2. 2

    Claude breaks it into contracted steps

    The spec becomes a detailed plan with 8–15+ steps, each with full contracts and verification criteria.

  3. 3

    Identify parallel work

    Steps that don’t share files or dependencies can run simultaneously in separate terminals.

  4. 4

    Launch with /agent-launch

    Start a wave — multiple terminals building independent features in parallel. Merge when they’re done. Can cut build time in half for large specs.

ⓘ
When to use Path C: When you have a complete specification and want Claude to execute it precisely. The directive ensures the spec is the authority, not Claude’s preferences.

From Idea to Build-Ready Plan

The three paths show how to start. This section covers the bridge between “I have an idea” and “Claude is building it.” Every feature — whether it began with /scope, a rough description, or a full spec — passes through the same planning process before any code is written.

Idea
→
/scope or describe
→
Plan (.claude/plans/)
→
Steps + Contracts

Build

1. Tell Claude what you want

You don’t need a formal spec. A few sentences describing what you want is enough. If it’s vague, run /scope — Claude asks you questions to pin down what you actually want. If it’s clearer, just describe it: “I want a page that shows X with Y, and the user can do Z.”

Too vague

“Build me a dashboard.”

Claude has to guess who it’s for, what data it shows, and what counts as done.

Better

“I want a dashboard that shows my 5 most recent sessions, with date, duration, and commit count. Include a chart of commits per day over the last 2 weeks.”

Claude knows the data, the visual elements, and the time window.

2. Claude writes a plan

Claude creates a plan file at .claude/plans/feature-name.md. The plan includes: what to build, how to build it, what order, and what model/effort level for each step. Plans are reference docs, not tracking docs — they live in .claude/plans/ and are read-only once building starts.

.claude/plans/recipe-search.md 
# Recipe Search — Plan

## Goal
Add search to the recipe list so users can find by name or category.

## Steps
1. Data layer — add search index to recipe data
   Model: Sonnet + low (mechanical)

2. Search UI — search bar, filter pills, results table
   Model: Sonnet + medium (spec-driven)

3. Integration — wire search input to filtered results
   Model: Sonnet + medium

4. Polish — loading states, empty states, mobile layout
   Model: Opus + high (design judgment)

5. Housekeeping — changelog, version bump
   Model: Sonnet + low (mechanical)

# Recipe Search — Plan

by name or category.

## Steps

2. Search UI -- search bar, filter pills, results
3. Integration -- wire search input to results
4. Polish -- loading states, empty states, mobile
5. Housekeeping -- changelog, version bump

3. How to review a plan

You don’t need to understand the technical details. Focus on these questions:

  • Does the end result match what I asked for? If the plan describes something different from what you imagined, say so now.
  • Does the step order make sense? Data before UI, basics before advanced features. If step 3 depends on step 5, that’s a problem.
  • Are there steps I don’t understand? Ask Claude to explain — “What does step 2 actually produce?”
  • Is anything missing? “What about mobile?” “What about error states?” “What happens when there are no results?”
⚠
This is the cheapest time to change direction. Before any code exists, a change costs one sentence. After three steps are built, a change might cost an hour of rework. Say something now if something looks wrong.

4. Plans become steps with contracts

From the plan, Claude creates numbered steps in tasks/todo.md. Each step gets a contract — testable criteria that define “done.” There are two types:

[auto] criteria

Claude verifies these by running a command. If the command passes, the criterion is met. No human needed.

[manual] criteria

You verify these with your eyes. Claude presents them as a checklist when it’s time for you to test.

2. [ ] Search UI — search bar, filter pills, results table → v0.5.1
   Contract:
   - [ ] [auto] Search input rendered. Verify: file: src/js/recipes.js contains recipe-search
   - [ ] [auto] Filter pills for categories. Verify: file: src/js/recipes.js contains filter-pill
   - [ ] [auto] Build succeeds. Verify: run: python3 execution/build.py
   - [ ] [manual] Search feels responsive. Results update as you type. Works on mobile.
ⓘ
In Plain English: A contract is a checklist stapled to every task. Before Claude can say “done,” it has to tick every box — and the boxes are specific enough that you’d know if it was lying.

5. Good contracts vs. bad contracts

The quality of your contracts determines how reliably Claude builds. A good contract tells you (or Claude) exactly what to check and what the right answer looks like. A bad contract could pass even if the feature is broken.

Bad contracts

[auto] It works. Verify: run: test

Too vague — what does “it works” mean?


[manual] Looks good.

No criteria to judge against.


No Verify: pattern at all.

Can’t be checked automatically.

Good contracts

[auto] Search input exists. Verify: file: src/recipes.js contains search-input

Specific, checkable, concrete.


[auto] Build succeeds. Verify: run: python3 execution/build.py

Clear pass/fail — it builds or it doesn’t.


[manual] Results update within 1 second. Filter pills narrow to matching category only. Clearing search restores full list.

Specific enough you know exactly what to test.

6. Promoting a feature from roadmap to current

Features start as ideas in ROADMAP.md. When you’re ready to build one, it moves through a simple promotion flow:

ROADMAP.md
→
todo.md Queue
→
todo.md Current
→
Building starts

Only one feature sits in Current at a time. Finish what you started before starting something new. This keeps context focused and prevents half-built features from piling up.

7. The moment you say “go”

Once the plan is reviewed, contracts look right, and the feature is in Current — you’re ready. Run /stand-up to start the session. Claude reads todo.md, sees the feature in Current, and picks up the first unchecked step.

From here, Claude builds step by step, verifying contracts as it goes. You don’t need to manage the process — Claude follows the plan and contracts automatically. You’ll be called in for [manual] checks at the right moments, and everything else runs on autopilot.

Getting Better Results

The way you describe what you want directly affects what Claude builds. A vague prompt gets a vague result. A specific prompt gets something closer to what you actually need. But “specific” doesn’t mean “write a technical spec” — it means giving Claude enough context to either build the right thing or ask the right questions.

Before and After: Three Examples

Good prompts aren’t longer — they’re more specific about the things that matter.

Example 1: Authentication

Vague

“Build me a login.”

Claude guesses: basic form, no session persistence, unknown tech stack.

Specific

“I need user authentication for my recipe app. Users need to create accounts, sign in, and stay signed in between visits. I don’t know whether to use a third-party auth service or build it from scratch — what would you recommend?”

Claude now knows: what, where, what users need, and where you’re open to advice.

Why the specific one works: It tells Claude WHAT (authentication), WHERE (recipe app), WHAT THEY NEED (create account, sign in, persist sessions), and WHERE THEY’RE UNSURE (build vs. buy). Claude will now ask about password reset flows, OAuth providers, session storage — things you didn’t know to ask about.

Example 2: Dashboard

Vague

“Make a dashboard.”

Claude guesses what data to show, what time periods to use, where to find the data.

Specific

“I want a dashboard as the first thing I see when I open the app. It should show my 5 most recent sessions with dates and duration, a chart of commits per day over the last 2 weeks, and my current streak. The data is already in .claude/stats.json.”

Claude knows: layout, data source, what “recent” means, and which visual elements to build.

Why the specific one works: It points to the data source, specifies what “recent” means (5 sessions), and describes the visual elements. Claude doesn’t have to guess at any of these.

Example 3: Bug Report

Vague

“It’s broken.” or “The search isn’t working.”

Claude has to search the whole codebase for the problem.

Specific

“When I type in the search box on the recipes page, nothing happens. Results don’t filter. It was working yesterday before we added the category filters. The search input is in src/js/recipes.js.”

Claude can jump straight to the problem: symptom, when it broke, what changed, where to look.

Why the specific one works: It describes the symptom, when it broke, what changed, and where to look. Claude can jump straight to the problem instead of searching the whole codebase.

Six Rules for Better Prompts

  • 1
    Say what it’s FOR, not just what it IS. “A chart showing voter turnout” is okay. “A chart showing voter turnout so canvassers can see which areas have low engagement” is better — now Claude knows the purpose and can make design decisions that serve it.
  • 2
    Tell Claude what you DON’T know. “I’m not sure if this should be a separate page or a tab on the existing page” gives Claude permission to recommend rather than guess. Claude is good at trade-offs when you tell it you’re open to options.
  • 3
    Point to existing things. “Make it look like the Pulse page” or “Use the same card layout as the targeting table” gives Claude a reference. It can read those files and match the pattern.
  • 4
    Give Claude the constraints. “This needs to work offline” or “We can’t use any paid APIs” or “This has to load in under 2 seconds” shapes the solution before Claude starts building.
  • 5
    Don’t over-specify HOW. Describe the outcome, not the implementation. “I want users to be able to save recipes for later” is better than “Add a button that writes to localStorage with a JSON array of recipe IDs.” You hired Claude for the technical decisions — let it make them.
  • 6
    If Claude builds the wrong thing, be specific about what’s wrong. “The chart should be horizontal, not vertical” or “I meant filter by category, not search by name.” Vague corrections (“that’s not right”) waste a round-trip.
ⓘ
Think of it like briefing a contractor. “Build me a shed” gets you something, but “I need a garden shed that fits my lawnmower and bike, with a door wide enough for the mower, and I’d like it to match the house colour” gets you what you actually want. You don’t need to know how to build the shed — you just need to describe the result clearly.

Feature Lifecycle

Regardless of which path you start from, every feature follows the same lifecycle. The path determines how you enter the pipeline — the pipeline itself is always the same.

Idea
→
Plan
→
Queue
→
Current
→
Build
→
Verify
→
Ship
→
Sign-off
→
Done

Stage 1 — Idea

Every feature starts as an idea. It might be yours, or Claude might pitch it while working on something else. If it’s vague, run /scope to clarify it through conversation. If it’s clear, describe it directly and ask Claude to plan.

Stage 2 — Plan

Claude writes a plan to .claude/plans/ with sequenced steps, technical approach, and the right model/effort level for each step. From this plan, Claude creates steps in todo.md (each with a contract) and a ROADMAP.md entry.

Stage 3 — Queue

Steps sit in the Queue section of todo.md until it’s time to build. The roadmap entry shows the feature under “Up Next.” Nothing is built yet.

Stage 4 — Current

When work begins, the feature moves from Queue to Current in todo.md. Claude is now actively building it. Only one feature lives in Current at a time.

Stage 5 — Build

Each step follows the cycle: build → verify → fix → commit. Claude writes code, runs the [auto] contract checks, fixes any failures (up to 3 attempts), and commits each passing step to git. Version numbers bump with each step.

Stage 6 — Verify

For large features (5+ steps), Claude pauses after the core UI step for a mid-feature checkpoint — presenting accumulated [manual] checks for you to test. For smaller features, all manual checks are presented at the end. This is your chance to catch visual bugs before dependent steps build on top of them.

Stage 7 — Ship

A housekeeping step handles the changelog entry, final version bump, and roadmap update. At feature completion, a pull request is created to merge the feature branch back to main, providing a review checkpoint before changes reach the stable branch. This keeps your project’s records clean and accurate.

Stage 8 — Awaiting Sign-off

The feature is now code-complete — all steps are built and all [auto] criteria pass. The feature moves to the Awaiting Sign-off section in todo.md immediately. Claude presents the full manual test checklist for you to work through.

⚠
The feature stays in Awaiting Sign-off until you test it. Claude cannot verify things that need human eyes — layout, interaction feel, print rendering. Once you confirm all [manual] items pass, the feature moves to Done in todo.md and Complete in ROADMAP.md.

Stage 9 — Retro

After sign-off, Claude runs a retrospective: what worked, what was slow, what to do differently. Learnings get recorded to learnings.md, becoming part of the project’s institutional memory. The next feature starts in better shape because of this one.

ⓘ
Every feature follows this path. Small features move through the pipeline in a single session. Large features span multiple sessions, with /stand-up and /wrap keeping state intact between them.

Manual Testing & Sign-off

When a feature reaches Awaiting Sign-off, all code is written and every [auto] criterion passes. What remains are the [manual] items — things that need human eyes. The /snagging command turns these into an interactive test checklist you can work through systematically.

When It Triggers

After the final build step completes and all [auto] contract criteria pass, Claude checks whether any [manual] items remain. If they do, you run /snagging to generate the test checklist. This is the bridge between “code complete” and “actually done.”

The /snagging Command

Running /snagging reads your todo.md, extracts all [manual] contract items for the current feature, and generates a standalone HTML checklist at docs/[feature-name]-manual-tests.html.

Terminal — /snagging 
$ /snagging

Reading todo.md... found 8 [manual] items across 3 steps
Splitting multi-sentence items into individual checks...
Detecting console commands and setup prerequisites...
Tracing code for known edge cases...

Generated: docs/user-dashboard-manual-tests.html
  14 individual checks across 3 sections
  2 console commands with copy buttons
  1 known bug flagged from code trace

Open in your browser alongside the app to begin testing.

The Test Checklist

The generated HTML page is a self-contained testing tool. Open it in your browser alongside the app and work through the checks.

ℹ
Checklist features: Three-state toggles (untested / pass / fail) with square checkboxes that cycle on click. Failed items expand a notes field for recording what went wrong. Console commands appear in copyable code blocks. Each section has setup prerequisites and amber “Restore” callouts between sections to reset state. A timer tracks total test duration. Progress persists in localStorage so you can close the tab and resume later. Export buttons let you copy full results or failures only.

The Feedback Loop

The testing workflow is a loop, not a one-shot process:

  1. 1

    Test

    Open the checklist and work through each item. Mark checks as pass or fail. Add notes to failures describing what you saw.

  2. 2

    Export

    Click Copy Test Results (or Copy Failures Only if everything else passed). This copies a structured summary to your clipboard.

  3. 3

    Paste & Fix

    Paste the results back into Claude. Claude reads the failures, fixes the underlying code, and commits the fixes.

  4. 4

    Regenerate

    Run /snagging again to get an updated checklist reflecting the fixes. Known bugs update automatically based on code analysis.

  5. 5

    Repeat

    Continue the loop until all items pass. Once they do, the feature moves from Awaiting Sign-off to Done.

Writing Good [manual] Items

The quality of your test checklist depends on how well the [manual] contract items are written. A few guidelines:

  • One testable action per sentence. “Click the search bar, type a query, and verify results filter in real time” is three checks. Split them so each can pass or fail independently.
  • Specify the expected result. Not “check the chart looks right” but “bar chart shows 5 bars, tallest bar is labelled ‘London’.” If you can’t describe what “right” looks like, the check isn’t testable.
  • For console tests, describe what to verify — not what to type. The generator detects console commands automatically. Write “print output shows 12 rows with no duplicates” rather than “run python3 execution/export.py and check the output.”
ⓘ
Prefer [auto] over [manual] where possible. DOM presence checks, data flow assertions, and function call verification can all be [auto] criteria with Verify: patterns. Reserve [manual] for things that genuinely need human judgement — visual layout, interaction feel, print rendering.
