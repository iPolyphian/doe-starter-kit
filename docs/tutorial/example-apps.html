Example Apps

See DOE in action with four different projects — each starting from a different level of preparation.

Each example starts from a different place — from having no plan at all to having a full spec with a phased roadmap. Together, they show how the same DOE framework adapts to any level of preparation.

Recipe Book

"Making it up as I go"
Type: Personal web app
Starting point: A single sentence
Sessions: 2
Key concepts: /scope, contracts, STATE.md

The idea

"I want an app where I can add recipes, search them, and mark favourites. Maybe categories too."

No spec, no wireframes, no bullet points. Just a loose vision of what the end result should feel like. This is the most common starting point.

How you'd start

With an idea this vague, the first move is /scope. Claude asks questions to turn the fuzzy idea into a concrete brief before a single line of code is written.

  1. 1

    Run /scope

    Claude asks clarifying questions: Who's using this? Web app or local? Fixed categories or custom tags? What does a recipe look like?

  2. 2

    Review the brief

    Claude writes a one-page brief to ROADMAP.md. A clear target instead of a vague idea. Five minutes of scoping saves hours of rework.

  3. 3

    Claude creates a plan

    Five steps: data model, add recipe form, search, favourites and tags, polish. Each step gets a contract with [auto] and [manual] criteria.

  4. 4

    Build and wrap

    Work through the steps. Run /wrap at the end of each session to save your position and decisions to STATE.md.

Fitness Tracker

"I have a rough idea"
Type: Personal web app with charts
Starting point: Bullet points
Sessions: 3
Key concepts: Multi-session memory, learnings.md, data/UI separation

The idea

"Track daily workouts — type, duration, notes. Show weekly and monthly charts so I can see trends. Set goals and track streaks."

More concrete than a vague wish. Clear nouns (workouts, charts, goals, streaks) and clear actions (track, show, set). But still a list of wishes, not a buildable plan.

How you'd start

Give Claude the bullet points directly. Since the idea is reasonably clear, /scope isn't needed. Claude creates a six-step plan separating data from UI — a core DOE principle.

ⓘ
Why separate data from UI? If you later change how workouts are stored, only the storage layer changes. The charts stay untouched. If you change the chart library, the data layer is unaffected. Each layer has one reason to change.
  1. 1

    Claude plans: data first, UI second

    Steps 1–2 are the storage layer. Steps 3–6 are the interface. This ordering is deliberate — verified data before building views on top of it.

  2. 2

    Session 1: data model and storage

    A date range query test fails — off-by-one error (< instead of <=). Claude diagnoses and fixes it. Two commits, then /wrap.

  3. 3

    Session 2: input form and history

    Claude discovers workout dates stored in UTC display wrong in local time — a workout logged at 11pm Monday shows up under Tuesday. Fixes the date handling and logs the bug to learnings.md.

  4. 4

    Session 3: charts and goals

    The stand-up card shows the timezone learning from Session 2. Claude uses timezone-aware dates correctly from the start — no repeat of the same mistake. The project got smarter.

DOE Concepts in Action
  • Data/UI separation — storage layer and chart display are independent; fixing one doesn't break the other
  • Contract verification — chart tests verified the chart renders with real data, not just that code exists

  • Session continuity — each session started exactly where the last left off via STATE.md and todo.md

Local Events Board

"I have a detailed spec"
Type: Web app with external API and maps
Starting point: Written specification
Sessions: 3
Key concepts: Directives, execution scripts, progressive disclosure, self-annealing

The idea

"Pull events from the Eventbrite API. Show them on a Leaflet map. Filter by category and date range. Click an event marker to see details."

A proper specification. Named data source, named technology, specific features, clear end result. When you start with this much clarity, DOE can move fast.

How you'd start

The spec becomes a directive in directives/ — the authority Claude follows throughout the build. The spec says "Leaflet map" — that means Leaflet, not a substitute Claude prefers.

  1. 1

    Write the directive

    Copy the spec into directives/events-board.md. Add a Progressive Disclosure trigger in CLAUDE.md: "before touching the events import, read the directive."

  2. 2

    Claude creates an 8-step plan

    Starting with the critical DOE moment: a dedicated execution script for the API import. No inline API calls in conversation.

  3. 3

    Session 1: build the import script

    Claude builds execution/import_events.py. Before writing code, it checks learnings.md for existing API patterns — progressive disclosure in action. Script handles auth, pagination, caching, validation, dry-run flag.

  4. 4

    Session 2: Leaflet map

    The directive says "Leaflet" — Claude follows the spec, not its own preference. Event markers positioned from the JSON output of the import script.

  5. 5

    Session 3: filters and detail panel

    A null category crashes the filter. Self-annealing: Claude reads the error, fixes the filter, adds a validation rule to the import script, logs the finding to learnings.md. One bug made the system stronger.

DOE Concepts in Action
  • Directives — the spec is the authority, not Claude's preferences; "Leaflet map" means Leaflet, not a substitute
  • Execution scripts — deterministic Python script does the same thing every time; no AI reasoning in the data path
  • Progressive disclosure — API patterns from learnings.md loaded automatically when the import script was started
  • Self-annealing — a bug became permanent knowledge; the import script gets more robust with every failure

Team Dashboard

"Full spec with roadmap"
Type: Multi-user app with auth and roles
Starting point: Full spec + phased roadmap
Sessions: 10+
Key concepts: Multi-phase builds, parallel waves, governed docs, compounding learnings

The idea

"Team dashboard with three roles: admins see everything, managers see their team's data, members see their own. Authentication via email invite. Real-time updates when data changes."

A serious project with users, roles, authentication, and real-time data. Building this without structure would be chaos.

How you'd start

The roadmap is written in three phases before building starts. Claude translates it into 12+ contracted steps. A governed security policy directive constrains how auth is built.

Phase 1: Foundation

Auth, user management, roles. Sessions 1–4. A governed security directive constrains every decision.

Phase 2: Views

Admin, manager, member dashboards. Sessions 5–6. Independent views run in parallel with /agent-launch.

Phase 3: Polish

Real-time updates, notifications, onboarding. Sessions 7–10. Compounding learnings from Phases 1–2 make this dramatically faster.
