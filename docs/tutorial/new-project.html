Starting a New Project

Use the DOE init wizard to scaffold a new project from scratch, or add DOE to an existing codebase. The wizard detects your framework, asks a few questions, and generates everything you need.

Prerequisites

Before running the init wizard, make sure you have:

Python 3

The wizard is a Python script. Most Macs have it pre-installed. Check with python3 --version in your terminal.

Git

Version control for your code. The wizard initialises a git repo if one doesn't exist. Get it from git-scm.com.

Claude Code

The AI coding assistant that drives DOE. After the wizard finishes, you'll open Claude Code and type /stand-up to begin.

You also need a local clone of the starter kit. If you haven't done this yet, follow the Installation guide first.

ⓘ
Where does the kit live? The wizard is run via bash ~/doe-starter-kit/setup.sh from inside your project directory. The kit itself stays in ~/doe-starter-kit/ and is never modified — it copies files into your project.

New Project

Starting from an empty directory is the simplest path. The wizard asks all the questions and sets everything up.

  1. 1

    Create your project directory

    mkdir my-app && cd my-app

    Replace my-app with whatever you want your project to be called.

  2. 2

    Run the wizard

    bash ~/doe-starter-kit/setup.sh

    The wizard walks you through a series of cards in your terminal. Each card asks one question. Here's a full walkthrough for a Next.js web app.

Framework init comes first. For a new project, the Done card tells you to run the framework's own scaffolding command (like npx create-next-app .) before launching Claude Code. DOE files sit alongside your framework's files and don't conflict.

Existing Project

If you already have code in the directory, the wizard adapts. It detects your framework from marker files and skips the questions it can answer automatically.

How detection works

The wizard checks for these files in your project directory:

  • package.json containing "next" → Next.js
  • package.json containing "vite" → Vite/React
  • requirements.txt or pyproject.toml → Python
  • go.mod → Go
  • pubspec.yaml → Flutter
  • index.html → Static HTML

When a framework is detected, Cards 1 and 2 (project type and framework) are skipped entirely. The wizard shows you what it found and asks you to confirm.

Your code is safe. The wizard never overwrites existing files. It only adds DOE files (CLAUDE.md, directives/, etc.) alongside your existing source code and configuration.
ⓘ
Wrong detection? If the wizard detects the wrong framework, enter n at the confirmation prompt to manually choose a different one. This is rare — detection is based on your package.json dependencies, not file names.

What Gets Installed

The wizard installs files in layers. Every project gets the universal base. Additional files are added based on your answers to the wizard questions.

Layer When active What it adds
Universal Always CLAUDE.md, directives/, execution/, .github/workflows/, .githooks/, tasks/todo.md, STATE.md, learnings.md, SECURITY.md, .doe-version
Public-facing Web apps, static sites Chrome verification directive for visual testing of web UI
Data handling "Store user data?" = yes CLAUDE.md rules for .env credential isolation and data import patterns
Regulated "Personal data?" = yes data-governance.md, legal-framework.md, DPIA template, GDPR compliance directives
ⓘ
Layers are additive. A web app that stores personal data gets all four layers: universal + public-facing + data handling + regulated. A CLI tool with no database gets only the universal layer.
Full file list (universal layer)

Every project, regardless of type or framework, receives these files and directories:

  • CLAUDE.md — project instruction file (framework-specific content)
  • STATE.md — session state tracker (updated by /wrap)
  • learnings.md — project learnings log (grows over time)
  • SECURITY.md — security policy and vulnerability reporting
  • .doe-version — tracks what was installed and your config choices
  • directives/ — planning, building, delivery, and review SOPs
  • execution/ — deterministic verification scripts
  • tasks/todo.md — empty task list ready for your first feature
  • .github/workflows/ — CI pipeline and auto-rebase workflow
  • .githooks/ — pre-commit hooks for quality checks
  • .claude/commands/ — slash commands (/stand-up, /wrap, etc.)
  • .claude/settings.json — hooks and agent configuration

Generated CLAUDE.md

The most important output of the wizard is CLAUDE.md. Unlike a generic template, the wizard generates this file with framework-specific triggers, directory structure, and best practices based on your choices.

Framework-specific triggers

The trigger table in CLAUDE.md changes based on your framework. Here's how the same section looks for Next.js versus Python:

Next.js triggers (web app with database)

## Triggers
- Planning/scoping → directives/planning-rules.md
- Building/coding → directives/building-rules.md
- Delivering (retro, PR, sign-off, wrap) → directives/delivery-rules.md
- About to skip a guardrail → directives/rationalisation-tables.md
- Security-sensitive code → directives/security-rules.md
- [APP] feature → directives/chrome-verification.md
# ... plus context-management, self-annealing, serial-dispatch, break-glass

Python triggers (API, no database)

## Triggers
- Planning/scoping → directives/planning-rules.md
- Building/coding → directives/building-rules.md
- Delivering (retro, PR, sign-off, wrap) → directives/delivery-rules.md
- About to skip a guardrail → directives/rationalisation-tables.md
- Security-sensitive code → directives/security-rules.md
# ... plus context-management, self-annealing, serial-dispatch, break-glass

The methodology triggers (planning, building, delivery, etc.) are universal — every project gets them. The difference is layer-specific: a web app gets chrome-verification.md for visual testing, while a Python API does not. If you enable the regulated layer (personal data), both would also get data-compliance.md and documentation-governance.md.

Directory structure

The directory structure section also adapts. A Next.js project includes src/ and public/. A Python API includes app/ and tests/. The wizard populates this based on what the framework conventionally expects.

ⓘ
You can always edit it. The generated CLAUDE.md is a starting point, not a locked file. As your project evolves, add triggers, remove ones that don't apply, and refine the rules. Claude reads this file at the start of every conversation, so keep it focused.

Next Steps

After the wizard finishes, you're ready to start building.

  1. 1

    Initialise your framework (new projects only)

    If you started from an empty directory, run the framework's scaffolding command. The Done card tells you which one — for example, npx create-next-app . for Next.js or pip install fastapi uvicorn for Python.

  2. 2

    Open Claude Code

    claude

    Launch Claude Code from your project directory. It reads CLAUDE.md automatically.

  3. 3

    Run your first stand-up

    /stand-up

    This reads your project state, initialises STATE.md, and shows a status card. On a brand-new project most fields will be empty — that's normal. You're ready to start building.

ⓘ
Want the full walkthrough? Head to Your First Session for a step-by-step guide from opening the terminal to wrapping up with all your progress saved.
ⓘ
Team mode later. If you chose solo and later need team features (code review assignments, security owner, multi-agent coordination), re-run the wizard with bash ~/doe-starter-kit/setup.sh --team. It upgrades in place without touching your existing files.
