DOE Architecture

Understanding the three-layer system that keeps AI-assisted development organised.

The Three Layers

DOE stands for Directive, Orchestration, Execution. It separates your project into three layers, each with a specific job. The AI handles reasoning (where it excels). Deterministic code handles mechanical tasks (where AI is unreliable). This separation is the foundation of everything DOE does.

Directive

Plain-English instructions stored in directives/. Like SOPs for how to handle specific tasks. No code here — just written intent.

Orchestration

Claude itself. Reads the directives, decides what to do, calls execution scripts in the right order, and handles errors.

Execution

Deterministic Python scripts for API calls, data transforms, file operations. These run the same way every time — no hallucination risk.

In Plain English

Directives are written instructions. Claude reads them and decides what to do. Execution scripts do the actual work. The separation means the AI handles the thinking, deterministic code handles the doing — no hallucinated file operations.

Directives

Directives are plain-English Markdown files in the directives/ folder. Each one is an SOP (Standard Operating Procedure) for a specific type of task — how to handle data imports, how to deploy, how to write tests, etc.

They define goals, inputs, tools to use, expected outputs, and edge cases. Claude reads the relevant directive before starting a task, so it knows the rules and constraints without you having to repeat them every session.

Directives are written by humans. They don't contain code — just clear instructions. When a directive is wrong or incomplete, you update it, and every future session automatically follows the new rules.

# directives/your-feature.md
# (Example: you create directives like this for your own project tasks)

## Goal
Import results from an external API into src/data/

## Inputs
- API endpoint (in .env as API_URL)
- Table or resource ID passed as argument

## Process
1. Run execution/import_data.py with the resource ID
2. Validate the output CSV has expected columns
3. Log row count to learnings.md if significantly different from last import

## Edge Cases
- API returns paginated results: script handles this automatically
- Rate limiting: script retries with exponential backoff

Execution Scripts

Execution scripts live in execution/ and are deterministic Python files. They handle the mechanical work that AI shouldn't improvise: API calls, data transforms, file operations, builds, and verification checks.

The key property is determinism: given the same inputs, an execution script produces the same output every time. There's no creativity, no judgment, and no hallucination risk. If a script worked yesterday with the same inputs, it will work today.

When no execution script exists for a task, Claude creates a new one following the patterns in existing scripts — same structure, same error handling, same logging. The new script becomes part of the toolkit for next time.

Contracts

A contract is a checklist attached to every task in tasks/todo.md. Each item is a specific, testable criterion — not vague ("looks good") but concrete ("the file exists and contains this function").

There are two types. [auto] criteria are verified by running a command or inspecting a file — Claude runs these itself after completing the work. If a check fails, it fixes the code and re-checks, up to three attempts. [manual] criteria need human eyes — does the layout look right? Does the interaction feel smooth?

In Plain English

A contract is a checklist attached to every task. Before Claude says "done", it runs each check to prove the work actually functions. This stops Claude from confidently claiming something works when it doesn't.

Here's what a contract looks like in tasks/todo.md:

### Step 3: Add search to recipe list
Contract:
- [x] [auto] Verify: file: src/recipes.js contains searchFunction
- [x] [auto] Verify: run: node tests/search.test.js
- [ ] [manual] Search feels responsive, results update as you type

When all auto criteria pass, the feature moves to Awaiting Sign-off — not straight to Done. Manual items are presented for you to test, and the feature only moves to Done once everything passes.

Key Files

CLAUDE.md

The project's constitution. Permanent rules, guardrails, directory structure, and triggers. Claude reads this at the start of every session.

STATE.md

Short-term memory. Current position, last session summary, blockers, and what to do next. Updated by /wrap at the end of each session.

learnings.md

Long-term memory. Patterns discovered, things that went wrong and how they were fixed. Grows over the project's lifetime.

tasks/todo.md

Active work tracker. Tasks, steps, contracts, and progress. The source of truth for what's been done and what's next.

Self-Annealing

Self-annealing is DOE's system for learning from failure. When something goes wrong, the framework doesn't just fix the immediate problem — it records what happened, why it happened, and how to prevent it next time. Over time, the project builds up immunity to its own failure patterns.

The name comes from metallurgy: annealing heats metal and cools it slowly to remove internal stresses. Each cycle makes the metal stronger. DOE does the same with software projects — each failure, properly recorded, makes the system less likely to fail that way again.

In Plain English

When something fails, the system doesn't just fix it — it writes down what went wrong and why, so the same failure can't happen again. Over time, the system accumulates immunity to its own mistakes.

A fresh DOE project has no learnings. Early sessions have more failures as patterns get discovered. But after 20 sessions, learnings.md contains the most common gotchas. After 50 sessions, Claude navigates the problem space with real expertise. The failure rate drops because the patterns are already known.

Routine failures get a one-line entry with a source tag:

- macOS sed requires empty backup arg: sed -i '' '...'. [retro: data-pipeline]
- Nomis API pagination starts at 0, not 1. [retro: census-import]

Significant failures (cost >30 minutes, broke something important, or recurred) get a structured entry with four parts: What happened, Root cause, Fix applied, Prevention added.

Thin Router (CLAUDE.md)

When a DOE project starts a session, Claude reads CLAUDE.md first. In early versions, this file contained every rule inline — over 100 lines of operating rules, guardrails, and instructions. The problem: Claude's context window is finite, and loading irrelevant rules wastes space and degrades performance.

The current design treats CLAUDE.md as a thin router — roughly 55 lines that define the project's identity, core behaviour rules, and a trigger table. The trigger table maps situations to directive files:

# Example triggers from CLAUDE.md
- Planning/scoping        -> directives/planning-rules.md
- Building/coding         -> directives/building-rules.md
- Delivering (retro, PR)  -> directives/delivery-rules.md
- Adversarial review      -> directives/adversarial-review/
- Something went wrong    -> directives/break-glass.md

When Claude encounters a planning task, it loads planning-rules.md. When building, it loads building-rules.md. Rules that aren't relevant to the current task never enter the context window at all.

In Plain English

Think of CLAUDE.md as a phone directory, not an encyclopaedia. It tells Claude where to find the right instructions for the current task. The instructions themselves live in separate files that only get loaded when needed. This keeps each session focused on what matters right now.

Phase-Based Directives

DOE groups its rules by workflow phase — what you're doing right now, not what type of rule it is. There are six phase-based directive files:

Planning Rules

Contracts, dependencies, scale-aware session planning. Loaded when scoping or starting a new feature.

Building Rules

Branch discipline, code hygiene, subagent protocol, file ownership. Loaded when writing code.

Delivery Rules

Retros, quality gates, guardrails, governance. Loaded when finishing or shipping work.

Context Management

Post-compaction recovery, solo vs parallel modes, shared-file awareness. Loaded after /clear or during parallel work.

Self-Annealing

Failure response, learnings curation, performance monitoring. Loaded when recording or reviewing learnings.

Framework Evolution

Tracking Claude Code platform changes, absorption decisions. Loaded when evaluating new platform features.

The benefit: when you're building, you get all the coding rules in one read — branch discipline, code hygiene, file ownership, quality gates. You don't get delivery rules or planning rules cluttering the view. Each phase gets exactly the context it needs.

Dependency Graphs (DAGs)

A DAG is a Directed Acyclic Graph — a map of steps connected by what actually depends on what. Instead of a numbered list where step 2 always follows step 1, a DAG identifies which steps are genuinely independent and can run simultaneously.

In DOE, each step in tasks/todo.md can declare two things:

  • Depends: — which other steps must finish before this one can start
  • Owns: — which files this step is allowed to modify
# From tasks/todo.md
0. [ ] Foundation -- Next.js + Tailwind setup
   Depends: none. Owns: package.json, tsconfig.json

1. [ ] App shell -- sidebar and topbar
   Depends: Step 0. Owns: components/shell/

2. [ ] Data layer -- JSON conversion + stores
   Depends: Step 0. Owns: lib/data/, lib/stores/

3. [ ] Dashboard page
   Depends: Steps 1, 2. Owns: app/page.tsx, components/dashboard/

The DAG executor reads this metadata and groups steps into waves. Steps 1 and 2 share no files and both only depend on Step 0, so they can run in parallel. Step 3 depends on both, so it waits. The executor handles this automatically — you describe the dependencies, the system figures out the parallelism.

In Plain English

Instead of doing steps one after another like a recipe, DOE looks at which steps are actually independent and runs them at the same time. "Build the sidebar" and "convert the data files" don't interfere with each other, so they happen simultaneously. "Build the dashboard" needs both, so it waits for them to finish.

Rationalisation Tables

AI agents are surprisingly good at constructing plausible reasons to skip guardrails. "This is too simple to test." "I'll add that later." "The contract is basically passing." These are rationalisations — convincing-sounding excuses that lead to real problems.

DOE includes rationalisation tables that map common excuses to their actual consequences. There are six tables covering TDD, verification, code review, contracts, scope creep, and trigger loading. Each row follows the same format:

The Excuse

"This is too simple to test — it's a one-liner."

Why It's Wrong

One-liners have edge cases. x.trim() fails on null. The test takes less time than this thought.

What To Do Instead

Write the test. If it's truly trivial, it takes 10 seconds. If it's not, you just caught a bug.

The 1% rule ties into this: if there's even a 1% chance a guardrail applies to your current task, load it. The cost of loading unnecessarily is a few hundred tokens of context. The cost of skipping is a bug that could have been prevented.

In Plain English

AI is good at convincing itself that rules don't apply. "Just this once" and "it's too simple to bother" are the most common paths to bugs. Rationalisation tables are a cheat sheet that maps these excuses to what actually happens when you follow them. When Claude catches itself thinking "this probably doesn't need...", it checks the table.

Adversarial Review

Self-review has a blind spot: the person who wrote the code tends to read what they intended, not what they actually wrote. DOE addresses this with adversarial review — a structured process where multiple AI agents review code from different angles with different incentives.

Three agents play distinct roles:

  • Finder — reads the implementation and identifies potential issues. Rewarded for real findings, penalised for false positives.
  • Adversarial — cross-examines the Finder's report against the actual code. Its job is to filter noise, not add findings. Penalised heavily for letting false positives through.
  • Referee — final arbiter for disputed findings. Scored on calibration accuracy.

The scoring incentives matter. The Finder is motivated to find real issues (not imaginary ones — false positives cost points). The Adversarial agent is motivated to be sceptical (letting a false positive through costs double). This creates a natural tension that surfaces genuine problems while filtering noise.

The level of review scales with risk. High blast-radius changes (shared files, API changes, security) get all three agents. Medium changes get Finder and Adversarial. Low-risk changes (docs, comments, config) can skip review entirely. The /review command triggers this process.

In Plain English

Instead of Claude reviewing its own work (and seeing what it wanted to see), three separate agents review it from different angles. One looks for problems, another challenges those findings, and a third settles disagreements. It's like having a code review team, except the reviewers are designed to disagree with each other.

Defence in Depth

No single safety measure is enough. DOE uses layered security where multiple independent mechanisms protect against the same class of failure. If one layer fails, the next catches it.

Layer 1: Hooks (Speed Bumps)

Pre-commit and pre-tool hooks catch common mistakes before they happen. Block secrets in code, warn about dangerous commands, enforce step marking. Fast, automatic, always running.

Layer 2: CI (Gates)

Continuous integration runs methodology checks, linting, and tests on every push. Catches issues that got past local hooks. Three tiers: gate (blocking), project (blocking), advisory (informational).

Layer 3: Human Review

CODEOWNERS ensures the right people review changes to critical files. PR-based workflow means no code reaches production without human approval.

Layer 4: Runtime (Last Resort)

Content Security Policy headers, input validation, and data isolation provide protection even if all other layers are bypassed. The final safety net.

The principle: deterministic beats probabilistic for enforcement. A directive that says "don't commit secrets" is probabilistic — the AI might forget. A pre-commit hook that scans for API key patterns is deterministic — it physically cannot be bypassed by forgetfulness. DOE pushes as many rules as possible into deterministic enforcement.

Deterministic Hooks

DOE converts rules into mechanical enforcement wherever possible. Instead of relying on Claude to remember a rule, a hook physically blocks the action if the rule isn't followed.

Examples from the DOE hook system:

  • Step-marking enforcement — a commit with "Step 3" in the message is blocked unless tasks/todo.md is staged with the step marked [x]. No bypass without an explicit skip flag.
  • Secret detection — pre-commit scans staged files for patterns that look like API keys, tokens, or passwords. Blocks the commit with a clear error message.
  • Review gate — PR creation is blocked on feature branches unless adversarial review artifacts exist. Forces review before shipping.
  • Sign-off routing — features with unchecked [manual] items are blocked from moving to Done. They must go through Awaiting Sign-off first.
  • Main branch protection — direct commits to main/master are blocked locally. Work must go through feature branches and PRs.

Each hook has a skip flag (e.g. SKIP_STEP_MARK_CHECK=1) for genuine edge cases, but the flags themselves are blocked by another hook if used on feature branches. The system is designed so that the path of least resistance is the correct path.

In Plain English

Rules that rely on someone remembering to follow them get forgotten. DOE converts its most important rules into automatic checks that physically prevent the wrong thing from happening. It's like a spell-checker that won't let you send the email until you fix the red underlines — except for code quality, not spelling.

Context Recovery

Long sessions eventually hit the context window limit. When this happens, Claude compacts the conversation — summarising earlier exchanges to free up space. This is necessary but lossy: details from earlier in the session, including loaded directives, can be forgotten.

DOE has a recovery protocol for this. After compaction (or after running /clear), Claude treats all directives as unloaded and re-reads the triggers relevant to the current task. CLAUDE.md includes explicit instructions for this:

# From CLAUDE.md Context Rules
After context compaction, treat ALL directives as unloaded.
Re-read triggers for your current task.
In wave/DAG mode: also re-read step assignment and ownership list.

This is why the thin router design matters. If all rules were inline in CLAUDE.md, compaction might lose some rules with no way to recover them (they'd be in the compacted summary somewhere, degraded). With the trigger table, recovery is simple: re-read the triggers, re-load the relevant directives. The directives are on disk, not in the conversation — they can always be re-read in full.

ⓘ
Prevention is better than recovery. Keep sessions focused on one task. Use /wrap to end a session cleanly when it's getting long. The Context & Sessions page covers this in detail.

AI Concepts You'll Encounter

DOE builds on top of Claude Code. To use DOE effectively, it helps to understand a few core concepts that affect how you work day to day. None of these require technical background — just a basic understanding of what's happening under the hood.

Context Window

Claude works within a context window — a limited amount of text it can see at once. Understanding how this works is essential for productive sessions. This is the single most important operational concept in AI-assisted development. Read the full guide: Managing Context.

Models

Claude comes in different versions (models) — Opus (most capable, slower), Sonnet (balanced), and Haiku (fastest, simplest). Some tasks need deep reasoning — architecture decisions, diagnosing subtle bugs. These benefit from Opus. Other tasks are mechanical — renaming files, running a known script. These work fine with Sonnet or Haiku.

DOE plan files recommend a model and thinking level for each step, so you know when to switch.

Tokens

The unit Claude reads and writes in. Roughly 4 characters equals 1 token. Both your input (what you say plus file contents) and Claude's output (responses plus code) consume tokens. Large files eat tokens fast — a 1,000-line source file might consume thousands of tokens just by being read.

DOE's execution scripts run outside the context window entirely — they don't cost tokens. Only the script's output comes back into the conversation.

Subagents

Claude can spawn helper "agents" to handle specific tasks. Each subagent gets its own context window and works independently, then reports back with a summary. This keeps the main conversation focused and prevents context window bloat. A subagent might read 10 files and generate 200 lines of code, but the main conversation only sees a brief summary: "Created the search component in 3 files, all tests pass."
