DOE Architecture

Understanding the three-layer system that keeps AI-assisted development organised.

The Three Layers

DOE stands for Directive, Orchestration, Execution. It separates your project into three layers, each with a specific job. The AI handles reasoning (where it excels). Deterministic code handles mechanical tasks (where AI is unreliable). This separation is the foundation of everything DOE does.

Directive

Plain-English instructions stored in directives/. Like SOPs for how to handle specific tasks. No code here — just written intent.

Orchestration

Claude itself. Reads the directives, decides what to do, calls execution scripts in the right order, and handles errors.

Execution

Deterministic Python scripts for API calls, data transforms, file operations. These run the same way every time — no hallucination risk.

In Plain English

Directives are written instructions. Claude reads them and decides what to do. Execution scripts do the actual work. The separation means the AI handles the thinking, deterministic code handles the doing — no hallucinated file operations.

Directives

Directives are plain-English Markdown files in the directives/ folder. Each one is an SOP (Standard Operating Procedure) for a specific type of task — how to handle data imports, how to deploy, how to write tests, etc.

They define goals, inputs, tools to use, expected outputs, and edge cases. Claude reads the relevant directive before starting a task, so it knows the rules and constraints without you having to repeat them every session.

Directives are written by humans. They don't contain code — just clear instructions. When a directive is wrong or incomplete, you update it, and every future session automatically follows the new rules.

# directives/your-feature.md
# (Example: you create directives like this for your own project tasks)

## Goal
Import results from an external API into src/data/

## Inputs
- API endpoint (in .env as API_URL)
- Table or resource ID passed as argument

## Process
1. Run execution/import_data.py with the resource ID
2. Validate the output CSV has expected columns
3. Log row count to learnings.md if significantly different from last import

## Edge Cases
- API returns paginated results: script handles this automatically
- Rate limiting: script retries with exponential backoff

Execution Scripts

Execution scripts live in execution/ and are deterministic Python files. They handle the mechanical work that AI shouldn't improvise: API calls, data transforms, file operations, builds, and verification checks.

The key property is determinism: given the same inputs, an execution script produces the same output every time. There's no creativity, no judgment, and no hallucination risk. If a script worked yesterday with the same inputs, it will work today.

When no execution script exists for a task, Claude creates a new one following the patterns in existing scripts — same structure, same error handling, same logging. The new script becomes part of the toolkit for next time.

Contracts

A contract is a checklist attached to every task in tasks/todo.md. Each item is a specific, testable criterion — not vague ("looks good") but concrete ("the file exists and contains this function").

There are two types. [auto] criteria are verified by running a command or inspecting a file — Claude runs these itself after completing the work. If a check fails, it fixes the code and re-checks, up to three attempts. [manual] criteria need human eyes — does the layout look right? Does the interaction feel smooth?

In Plain English

A contract is a checklist attached to every task. Before Claude says "done", it runs each check to prove the work actually functions. This stops Claude from confidently claiming something works when it doesn't.

Here's what a contract looks like in tasks/todo.md:

### Step 3: Add search to recipe list
Contract:
- [x] [auto] Verify: file: src/recipes.js contains searchFunction
- [x] [auto] Verify: run: node tests/search.test.js
- [ ] [manual] Search feels responsive, results update as you type

When all auto criteria pass, the feature moves to Awaiting Sign-off — not straight to Done. Manual items are presented for you to test, and the feature only moves to Done once everything passes.

Key Files

CLAUDE.md

The project's constitution. Permanent rules, guardrails, directory structure, and triggers. Claude reads this at the start of every session.

STATE.md

Short-term memory. Current position, last session summary, blockers, and what to do next. Updated by /wrap at the end of each session.

learnings.md

Long-term memory. Patterns discovered, things that went wrong and how they were fixed. Grows over the project's lifetime.

tasks/todo.md

Active work tracker. Tasks, steps, contracts, and progress. The source of truth for what's been done and what's next.

Self-Annealing

Self-annealing is DOE's system for learning from failure. When something goes wrong, the framework doesn't just fix the immediate problem — it records what happened, why it happened, and how to prevent it next time. Over time, the project builds up immunity to its own failure patterns.

The name comes from metallurgy: annealing heats metal and cools it slowly to remove internal stresses. Each cycle makes the metal stronger. DOE does the same with software projects — each failure, properly recorded, makes the system less likely to fail that way again.

In Plain English

When something fails, the system doesn't just fix it — it writes down what went wrong and why, so the same failure can't happen again. Over time, the system accumulates immunity to its own mistakes.

A fresh DOE project has no learnings. Early sessions have more failures as patterns get discovered. But after 20 sessions, learnings.md contains the most common gotchas. After 50 sessions, Claude navigates the problem space with real expertise. The failure rate drops because the patterns are already known.

Routine failures get a one-line entry with a source tag:

- macOS sed requires empty backup arg: sed -i '' '...'. [retro: data-pipeline]
- Nomis API pagination starts at 0, not 1. [retro: census-import]

Significant failures (cost >30 minutes, broke something important, or recurred) get a structured entry with four parts: What happened, Root cause, Fix applied, Prevention added.

AI Concepts You'll Encounter

DOE builds on top of Claude Code. To use DOE effectively, it helps to understand a few core concepts that affect how you work day to day. None of these require technical background — just a basic understanding of what's happening under the hood.

Context Window

Claude works within a context window — a limited amount of text it can see at once. Understanding how this works is essential for productive sessions. This is the single most important operational concept in AI-assisted development. Read the full guide: Managing Context.

Models

Claude comes in different versions (models) — Opus (most capable, slower), Sonnet (balanced), and Haiku (fastest, simplest). Some tasks need deep reasoning — architecture decisions, diagnosing subtle bugs. These benefit from Opus. Other tasks are mechanical — renaming files, running a known script. These work fine with Sonnet or Haiku.

DOE plan files recommend a model and thinking level for each step, so you know when to switch.

Tokens

The unit Claude reads and writes in. Roughly 4 characters equals 1 token. Both your input (what you say plus file contents) and Claude's output (responses plus code) consume tokens. Large files eat tokens fast — a 1,000-line source file might consume thousands of tokens just by being read.

DOE's execution scripts run outside the context window entirely — they don't cost tokens. Only the script's output comes back into the conversation.

Subagents

Claude can spawn helper "agents" to handle specific tasks. Each subagent gets its own context window and works independently, then reports back with a summary. This keeps the main conversation focused and prevents context window bloat. A subagent might read 10 files and generate 200 lines of code, but the main conversation only sees a brief summary: "Created the search component in 3 files, all tests pass."
