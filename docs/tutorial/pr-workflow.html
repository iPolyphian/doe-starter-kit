Workflows

PR Workflow

How features move from your branch to main through Pull Requests, CI checks, and snagging gates. Merging to main deploys to staging — production requires a separate promotion step.

What is a PR Workflow?

When you build a new feature, you do not work directly on the main copy of your project. Instead, you create a feature branch — a parallel copy of your project where you can build, experiment, and make mistakes without affecting the stable version that everyone relies on.

Once your feature is complete, you open a Pull Request (often shortened to PR). A Pull Request is a proposal to merge your changes from the feature branch back into the main branch. It is a formal checkpoint: "Here is what I built. Is it ready to go live?"

Before the merge happens, Continuous Integration (CI) runs automatically. CI is a set of automated checks — linting, testing, performance audits — that verify your code meets the project's quality standards. If any check fails, you fix the issue on your branch, push again, and CI re-runs. The main branch never sees broken code.

i
Think of it like an airlock. Your feature branch is the outer chamber. CI and snagging are the pressure checks. Only when everything passes does the inner door (main) open.

The DOE PR Cycle

Here is the full cycle from starting a feature to merging it into main. Every DOE project follows this pattern.

  1. 1

    Start with /crack-on

    The session command creates a feature branch from main. Your branch name follows the convention feature/feature-name. From this point, all your work happens on the branch — main stays untouched.

  2. 2

    Build step by step

    Work through your todo.md steps. Each step gets its own commit and push. This gives you a clear history and makes it easy to revert a single step if something goes wrong.

  3. 3

    Open a Pull Request

    At feature completion (the retro step), a Pull Request is created via gh pr create. The PR description summarises what changed, links to the plan, and lists any manual verification items.

  4. 4

    CI runs automatically

    As soon as the PR is created, Continuous Integration kicks in. It auto-detects your project's stack and runs the right checks. The CI Result check must pass before you can merge.

  5. 5

    Snagging checklist

    The snagging checklist verifies manual items that automated tests cannot cover — visual layout, interaction feel, print rendering. This is the final quality gate before merge.

  6. 6

    Merge to main

    When CI passes and snagging is complete, the PR merges. The feature branch is deleted. Main is always stable and always deployable. Merging here auto-deploys to staging — production is a separate manual step.

Feature Branches

A feature branch is a copy of your project that diverges from main at a specific point in time. You build your feature on this copy, and when it is ready, you propose merging it back. While you work on your branch, main remains exactly as it was — stable and unaffected.

Naming convention

DOE uses the pattern feature/feature-name. The name should be short and descriptive:

Terminal 
# Good branch names
feature/quality-stack
feature/pr-workflow-migration
feature/census-import

# Avoid vague names
feature/updates        # too vague
feature/fix            # what fix?

Why branches protect main

Without branches, every commit goes directly to main. If you are halfway through a feature and something breaks, main is broken for everyone. With a feature branch, broken code stays on your branch until you fix it. Main only receives complete, tested features.

i
One feature, one branch. Never build two unrelated features on the same branch. If one feature has problems, it blocks the other from merging.

Continuous Integration

Continuous Integration (CI) is a system that automatically runs your project's test suite every time you push code or open a Pull Request. You do not run these checks manually — CI does it for you, every time, consistently.

The three tiers

DOE CI auto-detects your project's stack (Node.js, Python, Go, Flutter, Rust, static HTML) and runs the right checks automatically. No configuration needed. Checks are organised into three tiers:

Gate (always runs)

Universal DOE checks: health check (stubs, TODOs, empty functions) and claim audit (documentation integrity). Works for any language. Blocks merge if it fails.

Project (auto-detected)

Stack-specific checks that only run when relevant files change. ESLint for JavaScript, pytest for Python, go test for Go. Blocks merge if it fails.

Advisory (informational)

Performance and browser checks like Lighthouse and Playwright. They run and report results, but never block your merge.

A single CI Result check aggregates all tiers. This is the only check branch protection requires. If you edit only markdown files, the gate passes and project-specific checks are skipped entirely.

Why three tiers?

Each tier exists for a specific reason:

  • Gate catches universal problems — broken documentation, stub functions left in code, TODO markers that should have been resolved. These apply to every project regardless of language or framework. If the gate fails, something fundamental is wrong and must be fixed before merging.
  • Project catches code-quality and correctness issues specific to your stack — a linting error in JavaScript, a failing unit test in Python, a type error in Go. These only run when you change relevant files, so editing a README does not trigger your entire test suite. If a project check fails, your code has a real problem that needs fixing.
  • Advisory tracks metrics you care about but should not block your work — Lighthouse performance scores, bundle size changes, test coverage percentages. A 2-point Lighthouse drop should not prevent you from shipping a critical bug fix. You see the results, you decide whether to act on them.
ⓘ
Advisory checks can be promoted. If your Playwright browser tests are stable and reliable, move them from advisory to project tier so they block merges. The tier system is a starting point — adjust it as your test suite matures. See the comments in doe-ci.yml for how to move checks between tiers.

When CI fails

If a gate or project check fails, the CI Result check fails and the PR is blocked from merging. This is intentional — it prevents broken code from reaching main. The fix is straightforward:

  1. Read the CI error output to understand what failed
  2. Fix the issue on your feature branch
  3. Commit and push the fix
  4. CI re-runs automatically on the new push
CI output 
# CI fails on ESLint
ESLint: 1 error
  src/app.js:42  'unusedVar' is defined but never used

# Fix the issue, commit, push
$ git add src/app.js
$ git commit -m "Remove unused variable"
$ git push

# CI re-runs automatically
All checks passed

Snagging as Pre-Merge Gate

Automated tests catch a lot, but some things need human eyes. Does the layout look right on mobile? Does the colour scheme work in dark mode? Does the print stylesheet produce clean output? These are snagging items — manual verification checks that run before merge.

The snagging checklist is generated automatically at feature completion. It combines two types of items:

  • Automated results: Pass/fail tiles showing the outcome of each CI check (gate, project-specific, and advisory). These are pre-filled — you do not need to re-run them.
  • Manual verification items: Checks that require a human to look at the result. Visual layout, interaction behaviour, content accuracy. You tick these off one by one.

Chrome visual verification

For visual checks, DOE provides the /chrome command. This opens the affected pages in Chrome so you can inspect them directly. No screenshots to compare — you look at the real rendered output and confirm it matches your expectations.

!
Do not skip snagging. Automated tests verify behaviour, but snagging verifies the experience. A page can pass every test and still look wrong. The snagging checklist is the last line of defence before your users see the result.

The Merge

When all CI checks pass and every snagging item is ticked off, the PR is ready to merge. Here is what happens:

  1. The feature branch's commits are merged into main
  2. The feature branch is deleted (it has served its purpose)
  3. Main now contains the new feature, fully tested and verified
  4. Anyone pulling from main gets the new feature immediately

The key principle: main is always stable. Every commit on main has passed CI and snagging. Main deploys to staging automatically — production only changes when you explicitly promote. You can deploy main at any time with confidence.

Terminal 
# PR merge via GitHub CLI
$ gh pr merge --squash --delete-branch

Merged PR #42: Add quality stack
Deleted branch feature/quality-stack

# Switch back to main and pull
$ git checkout main && git pull
Already up to date.
i
Squash merges keep history clean. Instead of adding every individual commit from your feature branch, a squash merge combines them into a single commit on main. The detailed history is preserved in the PR itself.

For Teams and Multi-Agent Workflows

The PR workflow scales naturally to teams and multi-agent setups. The principle is simple: each developer or agent gets their own feature branch. No two people work on the same branch at the same time.

This works because DOE tasks are designed to own distinct files. When Task A owns src/dashboard.js and Task B owns src/settings.js, their branches cannot conflict. Each PR merges cleanly because the changes do not overlap.

PRs merge sequentially — one at a time. If two PRs are ready simultaneously, the first merges, and the second rebases on the updated main before merging. This guarantees that main is always in a consistent state.

Solo developer

One feature branch at a time. Build, PR, merge, repeat. The simplest case.

Multi-agent wave

Multiple agents on parallel branches. Each owns distinct files. PRs merge sequentially after CI and snagging pass.
