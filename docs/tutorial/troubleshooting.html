Troubleshooting

When tools flag issues, this page tells you exactly what happened and how to fix it. Every scenario below shows what you'll see in your terminal, explains what it means in plain English, and gives you the exact command or action to resolve it.

ⓘ
How to use this page: Find the section that matches what you're seeing, then read the "What to do" row. You don't need to understand the internals — just follow the steps.

ESLint Issues

ESLint checks your JavaScript for problems before you commit. When it finds something, it tells you exactly which file and line to look at.

ESLint blocks my commit

What you see

An error in your terminal mentioning a filename and line number, like:

terminal 
src/js/main.js:42:5  error  'result' is assigned a value but never used  no-unused-vars
1 error found. Commit blocked.
What it means

ESLint found a code problem — in this example, a variable that's created but never actually used. The file name, line number, and rule name are all shown.

What to do

Read the error message and note the file and line number. Then tell Claude: "ESLint is flagging line 42 in main.js for unused variable 'result' — please fix it."

If you're in a hurry and need to commit anyway, you can bypass ESLint once:

SKIP_ESLINT=1 git commit -m "your message here"

Use this sparingly — it's a workaround, not a solution. The issue will still be there next time.

ESLint warns about unused variables

What you see

A list of warnings like:

terminal 
warning  'handleClick' is defined but never used  no-unused-vars
warning  'formatDate' is defined but never used  no-unused-vars
What it means

Functions are defined in a JavaScript file, but ESLint can't see that they're being called from your HTML (e.g. in onclick handlers or <script> tags). This is a common false alarm in projects that mix HTML and JS.

What to do

If the function is used somewhere in your HTML (like onclick="handleClick()"), it's safe to ignore the warning. Warnings don't block commits.

If you're confident the function is genuinely unused and not needed, you can delete it to keep the code tidy. When in doubt, ask Claude: "Is 'handleClick' actually used anywhere?"

I disagree with an ESLint rule

What you see

ESLint keeps flagging something you think is fine, or a rule is getting in the way of a deliberate design decision.

What it means

ESLint rules are configurable. The defaults are sensible but they're not always right for every project.

What to do

Tell Claude which rule is being flagged and why you want to change it. Claude will update eslint.config.mjs to disable or adjust the rule. For example: "ESLint keeps warning about 'no-console' but I need console.log for debugging — please turn that rule off."

Playwright Issues

Playwright runs automated browser tests — it opens a real browser and checks that your pages load correctly and behave as expected.

Playwright test fails

What you see
terminal 
FAIL  tests/dashboard.spec.js
  x dashboard loads correctly (2341ms)
    Error: locator('.summary-card') expected to be visible
What it means

A page didn't load correctly, or an element the test was looking for wasn't found. The test name tells you which feature is affected.

What to do

First, check if the feature works when you open it manually in your browser. If it looks fine to you, the test itself may be out of date or looking for a slightly different element — tell Claude: "The dashboard test is failing but the page looks fine in my browser. Please update the test."

If the page genuinely looks broken, that's a real bug. Ask Claude to investigate.

To see a visual report of what failed:

npx playwright show-report

Tests pass locally but fail in CI

What you see

Everything passes when you run tests on your machine, but they fail when GitHub Actions runs them.

What it means

There's an environment difference. CI runs on Linux with a specific screen size and no real display. Your Mac has different fonts, screen dimensions, and system behaviour. Timing-sensitive tests are also more likely to flake in CI because it's slower.

What to do

The most common culprits are viewport size and animation timing. Tell Claude: "Tests pass locally but fail in CI — the error is [paste error here]. Can you make the test more robust?" Claude can add explicit waits, fix viewport settings, or adjust the test to be less sensitive to timing.

Visual regression flags every page

What you see

Many screenshot comparison failures, often with messages like "screenshot differs from baseline".

What it means

Visual regression tests compare the current state of each page against a saved "baseline" screenshot. If the data on the page changes (new entries, updated charts), the screenshots won't match — even if the code is completely correct.

What to do

If the visual changes are intentional (you updated data, changed a chart, restyled a component), update the baseline screenshots so they reflect the new correct state:

npx playwright test --update-snapshots

Commit the new screenshots. If the changes look wrong or unexpected, that's a real regression — ask Claude to investigate.

Lighthouse Issues

Lighthouse measures how fast, accessible, and well-built your pages are. Scores run from 0 to 100 — higher is better.

Performance score dropped

What you see

A Lighthouse performance score lower than before, flagged in a test report or during a quality check.

What it means

New code made the app slower to load. This might be an expected trade-off (adding a large chart library will naturally slow things down), or it might be accidental (loading something twice, including a file that's not needed).

What to do

Ask yourself: did I just add a significant new feature? If yes, a modest score drop may be acceptable. If the drop is large (more than 10 points) or you didn't add anything obviously heavy, ask Claude: "The Lighthouse performance score dropped to [X] after my last change. What caused it and can we improve it?"

Accessibility score dropped

What you see

A lower accessibility score, often with specific failures listed like "image elements do not have alt attributes" or "buttons do not have accessible names".

What it means

A new UI element is missing something that screen readers and assistive technologies need — most commonly an alt attribute on an image, a label on a button, or insufficient colour contrast.

What to do

Ask Claude to fix the accessibility issue. Share the specific Lighthouse message: "Lighthouse is failing on 'buttons do not have accessible names' after I added the export button. Please fix it." This is almost always a quick fix — usually a single attribute.

Health Check Issues

The health check script scans your codebase for common problems — stubs, TODOs, and incomplete implementations.

Stub detected: return null

What you see
terminal 
[STUB]  src/js/api.js:18  fetchUserData() returns null
What it means

A function exists but doesn't do anything useful yet — it was written as a placeholder. Code that calls it will get null back and may behave unexpectedly.

What to do

Implement the function properly, or delete it if it's no longer needed. Tell Claude: "fetchUserData in api.js is returning null as a stub. Please implement it properly."

TODO found in shipped code

What you see
terminal 
[TODO]  src/js/charts.js:74  // TODO: handle empty data case
What it means

Someone (or Claude) left a reminder comment that never got actioned. The feature may work fine in normal cases but could fail when the situation described in the TODO arises.

What to do

Either do the TODO, or remove the comment if it's no longer relevant. Tell Claude: "There's a TODO at charts.js:74 about handling empty data — please implement it."

/code-trace Issues

The /code-trace command analyses your code and flags potential bugs, unreachable code, and unclear patterns. It uses three severity levels.

BUG: function returns NaN

What you see
terminal 
[BUG]  calculateTotal()  can return NaN when 'price' is undefined
What it means

NaN stands for "Not a Number". This happens when a calculation receives an undefined or non-numeric input. The function will silently return NaN instead of a number, which can cause strange behaviour (NaN is not equal to anything, including itself).

What to do

Claude will offer to fix it — say yes. The fix is typically adding a guard at the top of the function to handle the case where the input is missing.

WARN: unreachable code

What you see
terminal 
[WARN]  src/js/utils.js:31  code after return statement is unreachable
What it means

There are lines of code written after a return statement. A function stops executing as soon as it hits return, so anything after it can never run — it's dead code.

What to do

It's safe to delete the unreachable lines. Ask Claude: "There's unreachable code at utils.js:31 after a return statement. Please clean it up."

INFO: magic number

What you see
terminal 
[INFO]  src/js/charts.js:89  magic number: 86400
What it means

A number appears in the code with no explanation of what it represents. To someone reading the code later (including you), 86400 is meaningless without context. (In this case, it's the number of seconds in a day.)

What to do

This is an INFO — the lowest severity, and optional to fix. If you know what the number means, add a short comment next to it: // 86400 = seconds in a day. Or ask Claude to replace it with a named constant. You can also ignore it entirely.

npm / Node Issues

npm is the package manager that installs the tools your project depends on. Most npm problems have straightforward solutions.

npm install fails

What you see

An error during npm install, often with ENOENT, EACCES, or a long stack trace mentioning package names.

What it means

The packages couldn't be installed. This is often caused by a corrupted node_modules folder from a previous failed install, a lock file conflict, or a network issue.

What to do

Delete node_modules and try again from scratch:

rm -rf node_modules && npm install

If that still fails, check your Node version:

node --version

Then paste the full error to Claude for diagnosis.

npx command not found

What you see
terminal 
sh: npx: command not found
# or
npx: No such file or directory
What it means

The npm packages your project needs aren't installed yet, or Node itself isn't installed properly. npx comes with npm, so if it's missing, npm is missing too.

What to do

Make sure you're in your project folder, then run:

npm install

If that also fails with "command not found", Node is not installed. Follow the installation steps in the Getting Started guide.

Engine warnings during install

What you see
terminal 
npm warn EBADENGINE  Unsupported engine {
  package: 'some-package@3.0.0',
  required: { node: '>=20' },
  current: { node: 'v18.12.0' }
}
What it means

Your version of Node is older than a package recommends. This is a warning, not an error — the install will usually succeed and things will mostly work.

What to do

Usually you can ignore this and continue. If you start seeing strange errors or features not working, it may be worth upgrading Node to the latest LTS version. Ask Claude: "I'm getting engine warnings for Node v18. Do I need to upgrade for this project to work correctly?"

Git Hook Issues

Git hooks run automatically before you commit. They check for quality issues and can block the commit if something needs fixing.

Pre-commit hook blocks commit

What you see
terminal 
Pre-commit hook: ESLint found 2 errors. Commit blocked.
# or
Pre-commit audit: potential credentials found in src/config.js
What it means

The hook detected an issue — either an ESLint error or a potential security problem (like an API key in a file). These are important catches.

What to do

Fix the issue that was flagged. Ask Claude to help if needed. Once the issue is resolved, commit normally.

For genuine emergencies where you need to commit immediately and will fix the issue in the next commit, you can bypass all hooks:

git commit --no-verify -m "your message here"

Use this sparingly. The hook exists to protect you — bypassing it should be the exception, not the habit.

Contract check blocks commit

What you see
terminal 
Contract check: todo.md has unchecked [auto] criteria. Commit blocked.
What it means

Your todo.md has a task with acceptance criteria that haven't been verified yet. The contract check ensures you don't commit code that hasn't been tested against its own spec.

What to do

Run the verification command to check all the criteria:

/agent-verify

If you're doing housekeeping or documentation work that doesn't relate to any open task, you can bypass the check:

SKIP_CONTRACT_CHECK=1 git commit -m "your message here"

Snagging v2 Issues

The /snagging command runs a full quality sweep — Playwright tests, Lighthouse audits, accessibility checks, and visual regression — then presents results in a tile dashboard. These scenarios cover common failures you may see.

Lighthouse tile shows dash or error

What you see

The Lighthouse tile in the snagging dashboard shows a dash (-) instead of a score, or displays an error message:

terminal 
Lighthouse did not produce output
# or
Timed out waiting for Lighthouse
What it means

Lighthouse couldn't run at all. The three most common causes are:

  • Node version too old. Lighthouse 13 requires Node 20+. If you're on Node 18, it will silently fail.
  • Chrome not found. Lighthouse needs a Chromium-based browser to run its audits.
  • Dev server not running. Lighthouse needs something to audit on the expected port.
What to do

Check your Node version first:

node --version

If you're on Node 18, pin Lighthouse 11 which supports it:

npm install --save-dev lighthouse@11

If Node is fine, check that Chrome is installed and that your dev server is running on the expected port. The /snagging orchestrator starts the server automatically, but if you're running Lighthouse manually, you'll need to start it yourself.

All tiles show dashes

What you see

Every tile in the snagging dashboard shows - instead of results. No tests ran at all.

What it means

The dev server failed to start on port 8080. This usually happens because a stale process from a previous run is still occupying the port.

What to do

Kill whatever is using the port, then run snagging again:

lsof -ti:8080 | xargs kill

The snagging orchestrator normally handles this automatically, but if it couldn't kill the process (e.g. it was started by another user or requires elevated permissions), you may need to do it manually. After clearing the port, re-run /snagging.

APP_PATH mismatch warning

What you see
terminal 
WARNING: APP_PATH in test files references v2.3.0 but current version is v2.4.0
What it means

Your app version was bumped (e.g. the HTML file was renamed from app-v2.3.0.html to app-v2.4.0.html), but the test files still reference the old version. Tests will fail because they're trying to open a file that no longer exists.

What to do

The /snagging command auto-detects and fixes this. If you need to fix it manually, update the APP_PATH constant in each test file:

  • tests/app.spec.js
  • tests/accessibility.spec.js
  • tests/visual.spec.js

Change the version string to match your current app filename.

Visual regression shows unexpected diffs

What you see

The visual regression tile shows failures. Screenshot comparisons highlight differences between the current page and the saved baseline.

What it means

The page looks different from the last time baselines were captured. This could be an intentional UI change that hasn't been acknowledged yet, or it could be caused by non-deterministic data (timestamps, random IDs, live API responses) that changes between runs.

What to do

If the visual change is intentional (you restyled a component, changed a layout, updated data), update the baselines:

python3 execution/run_test_suite.py --update-visual

If the change is unexpected, investigate what's different. Common culprits are dynamic dates, randomised content, or a CSS change with unintended side-effects. Ask Claude: "The visual regression is flagging [page name]. I didn't change anything there — what's causing the diff?"

Dark mode toggle not working

What you see

The page is stuck in light mode or dark mode. Clicking the toggle button has no effect, or the page reverts to the wrong mode on reload.

What it means

The dark mode toggle stores the user's preference in localStorage. If a stale or corrupted value is stored, the toggle logic can get stuck — it reads the old value on page load and ignores clicks.

What to do

Clear the stored preference. Open your browser's DevTools (F12 or Cmd+Option+I), go to the Console tab, and run:

localStorage.removeItem('theme');
location.reload();

The exact key name may vary by project (e.g. 'dark-mode', 'color-scheme'). Check your app's toggle code if 'theme' doesn't work. After clearing, the toggle should work normally again.

Maestro Issues

Maestro runs automated mobile tests against an emulator or physical device. It's bootstrapped automatically by /snagging, but these issues can arise when the environment isn't set up correctly. See also: Testing & Quality > Getting Started with Maestro.

CLI not found

What you see
terminal 
maestro: command not found
What it means

Maestro isn't installed on your machine. The /snagging command auto-bootstraps it, but if you're running Maestro flows manually, you'll need to install it yourself.

What to do

Run /snagging, which handles installation automatically. Or install Maestro manually:

curl -Ls "https://get.maestro.mobile.dev" | bash

After installation, restart your terminal so the maestro command is on your PATH.

No devices found

What you see
terminal 
No devices found
What it means

Maestro can't find an emulator, simulator, or physical device to run tests against. It needs a running target before it can execute flows.

What to do

Start an emulator or simulator before running Maestro:

# Android — start an AVD emulator
emulator -avd <your-avd-name>

# iOS — open the Simulator app
open -a Simulator

If you're using a physical device, connect it via USB and ensure USB debugging is enabled (Android) or developer mode is on (iOS). See Testing & Quality > Configuration for setup details.

Flow timeout

What you see

A Maestro flow times out after 180 seconds without completing. The test hangs and then fails.

What it means

The app took too long to load, or Maestro couldn't find an element it was looking for. This usually means the app isn't launching correctly on the device, or a UI element has changed and the flow YAML references a selector that no longer exists.

What to do

Check three things in order:

  1. Is the app installed? Open the app manually on the device/emulator. If it's not there, install it first.
  2. Does appId match? Check config.json — the appId must match the app's bundle identifier exactly. Verify with: adb shell pm list packages | grep <name> (Android).
  3. Have UI elements changed? If the app loads but the flow can't find an element, update the selector in the flow YAML to match the current UI.

appId mismatch

What you see

Maestro launches the wrong app, or fails to launch any app at all. The flow immediately errors or interacts with an unexpected screen.

What it means

The appId in config.json doesn't match the bundle identifier of the app installed on the device. Maestro uses this ID to launch the correct app.

What to do

Find the correct bundle ID and update config.json:

# Android — list installed packages
adb shell pm list packages | grep <your-app-name>

# iOS — check in Xcode under General > Bundle Identifier

Copy the correct ID into the appId field in config.json, then re-run the flow.

No .maestro/ directory

What you see
terminal 
No .maestro/ directory found
What it means

The Maestro flow files haven't been created yet. The .maestro/ directory contains your test flows in YAML format, and it needs to exist before Maestro can run anything.

What to do

Bootstrap the test suite, which creates the directory and initial flow files:

python3 execution/run_test_suite.py --bootstrap

Bundle Size Issues

The test suite tracks your build output size and alerts you when it grows significantly. This catches accidental bloat — unminified code, duplicate dependencies, or large assets that slipped in. See also: Testing & Quality > Baselines.

Bundle size grew by more than 5%

What you see
terminal 
Bundle size growth: 142KB -> 168KB (+18.3%) — exceeds 5% threshold
What it means

Your build output is significantly larger than the saved baseline. The 5% threshold is designed to catch unintentional bloat — things like a new dependency you didn't mean to include, unminified code, or large images bundled into the output.

What to do

Ask yourself: did I just add a significant new feature or dependency? If yes, the growth is probably intentional — update the baseline so future comparisons use the new size:

python3 execution/run_test_suite.py --update-baselines

If the growth is unexpected, investigate what changed. Common culprits: a large library added as a dependency, images or fonts included in the build, or a minification step that stopped working. Ask Claude: "The bundle size jumped 18%. What's causing it?"

No baseline exists

What you see

The bundle size check runs but reports no comparison — there's no previous measurement to compare against.

What it means

This is your first run, or the baseline file (tests/baselines/bundle-size.json) was deleted. The test suite will automatically create a baseline from the current build, so this is informational only.

What to do

Nothing — the baseline is set automatically on first run. If you want to explicitly create or reset all baselines:

python3 execution/run_test_suite.py --bootstrap

Misleading growth percentage

What you see

A large percentage growth (e.g. +10%) on a small bundle that only grew by a few kilobytes.

What it means

Percentage-based thresholds can be misleading on small bundles. A 50KB → 55KB change is 10% growth but only 5KB in absolute terms — probably harmless. The 5% threshold is tuned for typical app bundles; very small projects will trigger it more easily.

What to do

Look at the absolute numbers, not just the percentage. If the actual increase is small (under ~20KB), update the baseline and move on. If you're seeing this frequently on a small project, that's normal — it becomes more useful as your bundle grows.

CI / GitHub Actions Issues

When you set up GitHub Actions to run your tests automatically, you may encounter environment differences between your machine and the CI runner. These scenarios cover the most common ones.

Missing Chromium in CI

What you see
terminal 
browserType.launch: Executable doesn't exist at /home/runner/.cache/ms-playwright/chromium-*/chrome-linux/chrome
What it means

Playwright's Chromium browser isn't installed on the CI runner. npm install installs the Playwright package but not the browser binary — that's a separate step.

What to do

Add a browser install step to your GitHub Actions workflow, after npm install and before the test step:

- name: Install Playwright browsers
  run: npx playwright install chromium

CI timeout

What you see

Tests pass locally but time out or flake in GitHub Actions. You may see Timeout exceeded errors or tests that intermittently fail.

What it means

CI runners are typically slower than your local machine. Default timeouts that are generous locally can be too tight in CI, especially for Lighthouse audits and screenshot comparisons.

What to do

Increase the timeout for CI runs. You can do this in playwright.config.js or pass it as a flag:

npx playwright test --timeout 60000

For Lighthouse, the test suite already uses a 60-second timeout. If that's not enough in CI, ask Claude to increase it in the workflow file.

Permissions error on scripts

What you see
terminal 
Permission denied: ./execution/run_test_suite.py
What it means

The script doesn't have execute permissions on the CI runner. File permissions can differ between macOS (where you develop) and Linux (where CI runs).

What to do

Use python3 explicitly instead of running the script directly:

# Instead of: ./execution/run_test_suite.py
python3 execution/run_test_suite.py

Or add execute permissions in your CI workflow: chmod +x execution/*.py

GitHub Actions caching

What you see

Every CI run re-installs all npm packages and Playwright browsers from scratch, making builds slow (3–5 minutes just for setup).

What it means

CI runners start fresh each time. Without caching, npm install and npx playwright install download everything on every run.

What to do

Add caching steps to your workflow. Ask Claude: "Add caching for node_modules and Playwright browsers to my GitHub Actions workflow." Claude will add the appropriate actions/cache steps, which typically cuts setup time to under 30 seconds.

Tests Pass but App is Broken

This is the trickiest situation you'll encounter: all automated tests pass (everything is green), but the feature doesn't actually work when you use it. Understanding why this happens — and what to do about it — is one of the most important lessons in software quality.

⚠
This is a learning moment, not a tool failure. When tests pass but the app is broken, the gap is in the contract — the specification of what "working" means — not in the testing tools themselves.

What happened

What you see

All automated tests pass. The snagging checklist shows green across the board. But when you actually use the feature in your browser, something is wrong — a button doesn't work, data is missing, a calculation is incorrect.

What it means

The contract criteria were too loose. Tests check exactly what was specified — nothing more. For example, a criterion like file: app.js contains handleClick verifies that the function exists, but it doesn't verify that the function actually works correctly. The function could have a bug, and the test would still pass.

This is not a flaw in the tools — it's the nature of automated testing. Tests can only verify what you tell them to check. See Testing & Quality > Baselines for more on how contracts and baselines work together.

What to do

Follow these four steps in order:

  1. Fix the bug. Get the feature working correctly first.
  2. Write a new auto criterion that would have caught the bug. For example, instead of just checking the function exists, write a Playwright test that actually clicks the button and verifies the result: 
    run: npx playwright test --grep "click handler saves data"
  3. Add it to the regression suite so the same bug can never happen again without being caught.
  4. Reflect on the gap. The problem was in the contract, not the tools. Ask: "What should I have specified to catch this?"
✓
The lesson: Tests are only as good as their contracts. When a bug escapes, the right response is to strengthen the contract — not to blame the testing tools. Every escaped bug is an opportunity to make the test suite smarter. Over time, your contracts get tighter and fewer bugs slip through.
